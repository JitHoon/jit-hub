# 3D GIS 지식 포트폴리오 — 프로젝트 블루프린트 v2

## 이 문서의 목적

이 문서는 **온톨로지 기반 지식 포트폴리오 웹사이트**의 종합 설계 지침이다. v1 블루프린트를 기반으로, Claude.ai 설계 세션(2026-03-29)에서 하나씩 검증·토론하여 고도화한 확정 버전이다.

### 두 가지 축

1. **온톨로지 기반 문제 해결**: 기술 노드의 의미적 관계를 네트워크 맵으로 시각화하는 지식 포트폴리오
2. **하네스 엔지니어링**: Claude Code의 skills, hooks, sub-agents를 활용해 에이전트 개발 환경을 구축하고, 이 경험 자체를 재사용 가능한 스킬셋이자 포트폴리오 콘텐츠로 만드는 것

## 1. 프로젝트 개요

### 누구의 프로젝트인가

- **이름**: 최지훈
- **경력**: 3D GIS/Cesium.js 특화 프론트엔드 개발자 2년차 (경력직 이직 준비 중)
- **개발 경험**: React, TypeScript, Cesium.js, Three.js, WebGL, Docker, AWS Lambda
- **목적**: 이직 면접용 지식 포트폴리오 사이트

### 핵심 욕구

- 자기 자신의 지식을 구조화하고 싶다
- 소비자 우선순위: 자신 = 면접관 = 다른 개발자 (동등)
- 최종 형태: 싱글 페이지 분할 뷰 기반 네트워크 맵 + 본문 읽기
- 콘텐츠는 지속적으로 성장한다 (21개에서 시작, 계속 추가 예정)
- 지식 노드 외에, AI로 빌드한 프로젝트 쇼케이스도 별도 섹션으로 추가 예정

### 면접관의 첫인상 목표

> "이 사람의 전문 영역이 한눈에 보이고, 하나씩 눌러보고 싶다"

## 3. 데이터 아키텍처

### 단일 소스 원칙 (Single Source of Truth)

**각 마크다운 파일의 프론트매터가 유일한 진실의 원천이다.**

- 새 노드 추가 시 `.md` 파일 하나만 만들면 됨
- `graph-data.json`은 빌드 타임에 모든 `.md` 프론트매터를 수집하여 **자동 생성**
- 자동 생성된 `graph-data.json`은 `.gitignore`에 추가
- 빌드 타임에 관계 무결성 검증 (존재하지 않는 slug 참조 시 빌드 에러)

### 프론트매터 스키마 (확정)

```yaml
---
slug: "node-slug"           # URL 경로, 파일명, 그래프 노드 ID
title: "노드 제목"
cluster: "geodesy"           # 9개 클러스터 중 하나
difficulty: "intermediate"   # beginner | intermediate | advanced | expert
prerequisites:               # 선행 노드 (이 노드를 이해하려면 먼저 알아야 하는 것)
  - "prerequisite-slug"
relatedConcepts:             # 관련 노드와 관계
  - slug: "related-slug"
    relationship: "appliedIn"  # 자유 확장 가능한 관계 유형
childConcepts:               # 하위 개념 노드
  - "child-slug"
tags: ["tag1", "tag2"]
---
```

### Zod 스키마

프론트매터 검증을 위한 Zod 스키마는 코드에서 정의하고 관리한다. 클러스터 목록, difficulty 레벨 등 구체적인 enum 값은 코드가 단일 소스이다.

### 클러스터 구조

- 클러스터 목록, 라벨, 색상은 코드에서 정의
- 각 노드가 어떤 클러스터에 속하는지는 해당 노드의 프론트매터 `cluster` 필드에 기록
- 클러스터 추가/변경 시 코드의 enum과 색상 정의만 수정하면 됨

### 관계 유형 2단계 시스템

- **데이터 레벨 (프론트매터)**: 관계 유형을 제한 없이 자유롭게 사용. 새 노드 추가 시 새 관계 유형 생성 가능.
- **시각 레벨 (그래프 렌더링)**: 상세 유형을 4~5개 시각 카테고리로 자동 매핑. 매핑 테이블은 코드에서 관리.
- 매핑에 없는 새 관계 유형은 기본 카테고리("관련")로 fallback
- 본문 패널에서는 원래의 관계 유형을 텍스트로 그대로 표시

### 노드 연결 티어 (허브 판정)

- 빌드 타임에 각 노드의 연결 수를 계산하여 허브/리프를 자동 분류
- 허브 판정 기준(연결 N개 이상)은 코드에서 설정 값으로 관리
- 노드가 늘어나면 허브/리프 분류가 자동 조정됨

> **폴더 구조**: 구체적인 파일 배치(lib/, src/ 등)는 Claude Code 환경에서 개발하면서 결정한다.

## 4. UX/인터랙션 설계

### 핵심 뷰: 싱글 페이지 분할 뷰

별도의 `/nodes/[slug]` 상세 페이지로 이동하지 않는다. 하나의 페이지에서 그래프와 본문이 나란히 존재한다.

#### 상태 1: 초기 화면 (그래프 전체 화면)

- 허브 노드 5개만 표시 (연결 5개 이상)
- 허브 노드 크기는 연결 수에 비례
- 클러스터별 색상으로 성격 구분
- 허브 간 관계선만 표시, 허브만 제목 표시
- 리프 노드는 숨겨진 상태

#### 상태 2: 허브 호버 → 리프 펼치기

- 호버한 허브의 직접 연결 리프 노드가 애니메이션으로 등장
- 다른 허브 호버 시 이전 리프 접힘 + 새 리프 펼침
- 호버 해제 시 리프 숨김
- 리프 제목은 호버 시에만 표시

#### 상태 3: 노드 클릭 → 분할 뷰

- 그래프가 왼쪽으로 축소, 오른쪽에 본문 패널 열림
- 좌우 비율은 사용자가 드래그로 조절 가능
- 왼쪽 그래프에서 다른 노드 클릭 시 오른쪽 본문만 전환
- 그래프 상태(줌, 펼침)가 유지됨

#### 모바일

- 본문이 그래프 아래로 펼쳐짐
- 그래프는 상단에 축소 표시

### URL/SEO 전략

두 가지 경로를 모두 제공:

- **`/`** — 분할 뷰. 쿼리 파라미터 `/?node=cesium-mouse-events`로 특정 노드 열기
- **`/nodes/[slug]`** — SEO용 정적 페이지. 구글 크롤링 대응. 사용자 접근 시 분할 뷰 상태로 열어줌

### /projects 섹션 (별도)

지식 그래프와 독립적인 프로젝트 쇼케이스 영역. 별도 스키마로 관리.
- MVP 이후 구현
- AI로 빌드한 사이트들의 링크/미리보기

## 5. 콘텐츠 파이프라인 상세

### 빌드 타임 처리 흐름

```
content/nodes/*.md (프론트매터 + MDX 본문)
        │
        ▼
  gray-matter로 프론트매터/본문 분리
        │
        ▼
  Zod 스키마로 프론트매터 검증
        │
        ├──▶ 프론트매터 데이터 → graph-data.json 자동 생성
        │                        (nodes, edges, clusters)
        │
        └──▶ MDX 본문 → remark/rehype 처리
                         → next-mdx-remote로 렌더링
                         → 커스텀 컴포넌트 삽입 가능
                           (3D 데모, 다이어그램, 차트)
```

### 빌드 타임 검증 항목

1. 모든 `prerequisites`의 slug가 실제 노드로 존재하는지
2. 모든 `relatedConcepts`의 slug가 실제 노드로 존재하는지
3. 모든 `childConcepts`의 slug가 실제 노드로 존재하는지
4. slug 중복이 없는지
5. 필수 필드(slug, title, cluster, difficulty, tags)가 빠지지 않았는지

검증 실패 시 빌드 에러로 표시.

### MDX 커스텀 컴포넌트 (계획)

본문 안에서 사용할 수 있는 인터랙티브 요소:
- Cesium.js / Three.js 3D 데모 뷰어
- 다이어그램 / 차트 시각화 컴포넌트
- 코드 블록 하이라이팅 (rehype-shiki 등)

## 6. 하네스 엔지니어링

### 설계 원칙

- **점진적 구축**: 완벽하게 설계한 후 시작하지 않는다. Claude Code에서 개발하면서 환경을 조율해 나간다.
- **과적합 방지**: 규칙이 너무 촘촘하면 Claude의 창의적 해결을 막는다. 목표와 제약만 주고, 단계별 지시를 강제하지 않는다.
- **CLAUDE.md는 조건부 목차**: 모든 것을 때려넣지 않고, "코딩할 때는 A를 읽고, 테스트할 때는 B를 읽어라"와 같은 역할.

### 하네스 구성 요소 (초기 버전)

**확정된 것:**
- CLAUDE.md: 프로젝트 규칙, 조건부 목차 (별도 파일로 제공)
- architecture.md: 시스템 구조, 데이터 흐름 (별도 파일로 제공)
- progress.md: 현재 진행 상황, 다음 할 일 (개발 시작 시 생성)

**폴더 구조 확정 후 할 것:**
- architecture.md를 도메인별로 분리 (콘텐츠/빌드, UI/인터랙션)
- 폴더별 CLAUDE.md 배치로 Claude Code가 자동으로 필요한 컨텍스트만 읽게 구성

**Claude Code에서 점진적으로 구축할 것:**
- 서브에이전트 (design-agent, frontend-agent, infra-agent 등)
- Skills (graph-visualization, content-pipeline, nextjs-patterns 등)
- Hooks (PreToolUse, PostToolUse, Stop 등)

구체적인 서브에이전트/스킬/훅 구조는 개발 과정에서 필요에 따라 설계한다. 이 경험 자체가 포트폴리오의 일부가 된다.

### 하네스 경험의 포트폴리오화

1. `.claude/` 디렉토리를 GitHub에 공개
2. 개발 완료 후 "하네스 엔지니어링" 노드를 지식 그래프에 추가
3. README에서 하네스 설계 과정과 교훈 설명
