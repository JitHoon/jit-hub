# 콘텐츠 업그레이드 가이드

> 출처: 90% 서류 합격률 백엔드 개발자 이력서 분석 (유튜브 스크립트)
> 핵심 원리를 이력서 → 기술 블로그 콘텐츠에 맞게 변환

## 이 문서를 받은 에이전트에게

이 문서는 `contents/nodes/*.md` 콘텐츠를 수정하거나 새로 작성할 때의 **품질 기준서**다.

### 절대 원칙: 기존 콘텐츠를 그대로 유지하는 것은 없다

모든 노드는 **전면 리라이팅** 대상이다. "이미 잘 쓰여 있으니 수치만 추가하면 된다"는 판단은 금지한다. 기존 콘텐츠가 기술적으로 정확하더라도, 이 문서의 S++ 품질 기준(아래)에 도달하지 못하면 구조·톤·깊이를 모두 재작성한다.

### S++ 품질 기준

A급 콘텐츠는 "기술적으로 정확하고 구조가 있는 글"이다. 이것으로는 부족하다. S++ 콘텐츠는 **읽는 사람이 이 글의 저자와 일하고 싶다고 느끼는 글**이다.

| 기준 | A급 (부족) | S++ (목표) |
|------|-----------|-----------|
| **도입부** | "X는 Y를 하는 기술이다" 정의로 시작 | 실무 에피소드(상황→긴장→질문→결과)로 시작. 첫 3줄에 독자의 관심을 잡는다 |
| **문제 인식** | 문제가 있었다고 언급 | 문제의 **비즈니스 임팩트**까지 서술. "고객사 시연 중 크래시", "메모리 10.5MB→105MB로 10배 급증" |
| **대안 분석** | 선택한 기술을 설명 | 최소 2~3개 대안을 **트레이드오프 표**로 비교하고, "우리 상황에서 X를 선택한 이유"와 "X가 안 맞는 상황"까지 명시 |
| **사고 과정** | 결론만 제시 | 처음에 잘못 의심한 것, 배제한 근거, 결정적 단서 등 **사고 흐름**을 단계별로 서술 |
| **결과** | "개선되었다" | Before/After 수치 + 측정 도구/조건 명시 + 한계점까지 솔직하게 |
| **내러티브** | 독립된 기술 문서 | 다른 노드와의 연결이 있고, 전체 블로그 스토리의 한 챕터로 읽힌다 |
| **교훈** | 기술적 요약 | 이 경험에서 얻은 **보편적 원칙** 한 줄. 독자가 자신의 상황에 대입할 수 있다 |
| **권위** | 자기 경험만 서술 | 적절한 곳에 공식 스펙, 학술 논문, 저명 엔지니어의 글을 인용하여 주장에 공신력 부여 |
| **가독성** | 정보는 있지만 읽기 피곤 | 한 문장 40자, 한 문단 3~5줄, 비교는 표, 코드 위에 설명. 소리 내어 읽어도 자연스럽다 |
| **마무리** | 요약으로 끝남 | 독자가 **즉시 실행할 수 있는 액션** 1줄로 마무리 |

### 작업 전 필수 읽기 순서

1. **3-8절 Gold Standard** — `fbxloader-memory-leak.md`를 먼저 읽고 품질 기준을 잡는다
2. **3-6절 표준 구조** — 노드의 섹션 순서와 각 섹션의 역할을 파악한다
3. **4장 글쓰기 스타일** — 금지 표현(4-4절), 자가 점검(4-9절)을 숙지한다
4. **6장 해당 노드** — 수정 대상 노드의 구체적 지시사항을 확인한다. **6장에 언급되지 않은 노드라도 위 S++ 기준에 맞춰 전면 리라이팅한다**
5. **2장 기밀 보호** — 작성 후 반드시 기밀 점검을 수행한다
6. **`raw/user-sources/`** — 소스 자료를 직접 읽어서 콘텐츠에 반영할 구체적 수치·에피소드·맥락을 추출한다

### 작업 후 필수 점검

- 4-9절 자가 점검 **14항목을 모두** 통과하는지 확인한다
- 특히 항목 10("면접관 질문 3개")과 항목 14("페르소나 일관성")를 반드시 확인한다
- **S++ 기준표의 10개 항목 중 하나라도 A급 수준에 머물러 있으면 재작성한다**

## 1. 이 가이드의 목적

이 블로그의 독자는 세 종류다:

1. **나 자신** — 다시 읽었을 때 "그때 왜 이 결정을 했지?"가 바로 떠올라야 한다
2. **채용 담당자 / 기술 면접관** — 이력서 링크로 블로그를 볼 때, 엔지니어링 사고 수준을 판단한다
3. **같은 문제를 겪는 개발자** — "이거 나도 겪었는데, 이 사람은 어떻게 풀었지?"

세 독자 모두를 만족시키는 글의 조건: **왜 이 문제가 중요했고, 어떤 선택지 중에서 왜 이걸 골랐고, 결과가 어땠는지**가 명확한 글.

### 1-1. 블로그 전체의 페르소나

> "자기소개 첫 장에서 컨셉이 정해졌고, 그 인격을 가졌다면 어떤 식으로 이력서가 정해져야 될지 방향이 잡힌다. 스스로한테 가스라이팅을 하는 것이다."

이력서에서 자기소개가 전체 톤을 결정하듯, 이 블로그도 **하나의 일관된 정체성**이 있어야 한다. 모든 글이 같은 사람이 쓴 것처럼 읽혀야 하고, 그 사람의 관심사와 사고방식이 글마다 드러나야 한다.

**이 블로그의 페르소나**:
"브라우저에서 대용량 3D 데이터를 안정적으로 렌더링하는 것에 집착하는 프론트엔드 엔지니어. 메모리 크래시를 직접 겪었고, 뷰어 엔진을 혼자 교체했고, GPU 메모리를 73% 줄였다. 모든 기술 선택에는 '왜 이것이고 다른 건 왜 안 되는지'를 묻는다."

**모든 글에 이 페르소나가 반영되어야 한다:**
- 좌표계 이론 글이라도 "이걸 몰랐을 때 모델이 지구 반대편에 렌더링됐다"는 실무 맥락이 도입부에 있어야 한다
- 알고리즘 설명 글이라도 "이 알고리즘을 선택한 이유와 선택하지 않은 대안"이 있어야 한다
- 순수 이론 글은 이 블로그에 존재할 이유가 약하다 — 반드시 실무 맥락과 연결되어야 한다

### 1-2. "안 쓴 것만 못한 콘텐츠"는 삭제하라

> "돌이켜 보면 아무 쓸모가 없었다. 프로젝트를 왜 하는지에 대한 의도가 명확하지 않다 보니, 이력서나 면접에서 이걸 풀 수가 없었다. 아예 과감하게 없애 버렸다."

모든 노드가 존재할 가치가 있는 건 아니다. 아래 기준에 해당하면 **삭제하거나 다른 노드에 병합**하는 것이 낫다:

- "이걸 왜 했는지"를 한 문장으로 설명할 수 없는 콘텐츠
- 문제(P)도 분석(A)도 결과(R)도 없이 기술 사전처럼 개념만 나열한 콘텐츠
- 다른 노드의 하위 섹션으로 충분히 흡수될 수 있는 얇은 콘텐츠
- 블로그 페르소나와 연결되지 않는 콘텐츠

## 2. 기밀 보호 원칙

### 절대 금지 항목

| 분류 | 금지 대상 | 예시 |
|------|-----------|------|
| **고객 정보** | 고객사 이름, 프로젝트명, 계약 조건 | "A건설사의 밀양 현장" ❌ → "특정 건설 현장의 드론 촬영 데이터" ✅ |
| **내부 인프라** | 서버 주소, AWS 계정 ID, API 키, 내부 도메인 | `s3://company-bucket/...` ❌ |
| **비즈니스 수치** | 매출, 고객 수, 계약 금액, 내부 KPI | "월 매출 X억" ❌ |
| **미공개 로드맵** | 출시 전 기능, 내부 전략, 경쟁사 비교 전략 | "다음 분기에 Y 기능 출시 예정" ❌ |
| **팀 내부 사항** | 팀원 이름, 조직 구조, 내부 의사결정 과정의 갈등 | "PM이 반대했지만..." ❌ |
| **소스 코드** | 회사 코드베이스의 직접 복사 | 프로덕션 코드 그대로 ❌ → 핵심 로직만 추출한 의사 코드 ✅ |

### 안전하게 쓸 수 있는 것

| 분류 | 허용 범위 | 작성법 |
|------|-----------|--------|
| **기술 스택** | 사용한 라이브러리/프레임워크 이름 | "CesiumJS", "Three.js", "AWS Lambda" — 공개 기술 |
| **성능 수치** | 상대값(%, 배수) 또는 일반화된 절대값 | "GPU 메모리 73% 절감" ✅, "75MB → 20MB" ✅ |
| **문제 상황** | 익명화된 맥락 | "고객사 시연 중 크래시" ✅ (고객사 특정 불가) |
| **아키텍처 결정** | 기술적 트레이드오프 분석 | "AWS Batch vs EC2 비교" ✅ |
| **파일 수/용량** | 데이터셋 규모 (프로젝트 특정 불가한 수준) | "3,561개 타일, 2.6GB" ✅ |

### 작성 시 자가 점검

1. **이 정보로 회사/고객사를 특정할 수 있는가?** → 특정 가능하면 익명화하거나 삭제
2. **이 코드가 프로덕션 코드의 직접 복사인가?** → 핵심 개념만 추출한 예시 코드로 재작성
3. **퇴사 후에도 이 글이 문제가 되지 않을까?** → 불안하면 빼라

### 소스 자료(`raw/user-sources/`) 처리 규칙

- "밀양 프로젝트" → "실제 운영 프로젝트" 또는 "드론 촬영 프로젝트"로 일반화
- "26-0205" 같은 내부 프로젝트 코드 → 삭제
- 수치(3,561개 타일, 2.6GB 등)는 기술 벤치마크로서 일반적이므로 사용 가능

## 3. PAR 구조

### 3-1. 정의

| 요소 | 이력서 | 블로그 | 차이점 |
|------|--------|--------|--------|
| **P**roblem | 1줄 압축 | 상황을 풀어서 서술 | 독자가 "나도 겪었다"고 공감할 수 있도록 **맥락**을 깔아야 한다 |
| **A**nalyze | 핵심 결정 1~2개 | 검토한 대안 전체 + 트레이드오프 | "왜 A가 아니라 B를 골랐는지" 독자가 따라올 수 있게 서술한다 |
| **R**esult | 수치 1줄 | Before/After + 측정 방법 | 수치뿐 아니라 **어떻게 측정했는지**까지 서술해야 신뢰가 생긴다 |

### 3-2. AI 시대에 Analyze가 더 중요해진 이유

> "레디스 캐싱 구현, 인덱스 적용, MSA 전환 — 이런 구현 레벨의 텍스트는 너무 많은 이력서에 다 들어가 있다. AI로 인해 구현 비용이 저렴해졌기 때문에 이력서로서의 가치를 잃어가고 있다."

블로그도 마찬가지다. "KTX2로 텍스처를 압축했다"는 누구나 쓸 수 있다. 차별화되는 건:

- 왜 ETC1S를 골랐고 UASTC는 왜 안 골랐는지
- 팀 규모가 작아서 AWS Batch 대신 EC2를 선택한 현실적 이유
- dispose()를 했는데도 메모리가 안 줄어서 Retained Size를 추적한 과정

**"무엇을 했다"가 아니라 "무엇 사이에서 고민했다"가 글의 가치를 결정한다.**

### 3-3. "앞단의 사고 과정"을 드러내라

> "면접에서 공통적으로 받은 질문 — 단순히 개쩌는 문제 해결이 궁금한 게 아니라, 앞단에서 어떤 갈등과 결정 상황이 있었는지가 궁금한 것이다."
> "그럴듯해 보이는 문제와 그럴듯해 보이는 해결이 있어도, 어떤 사고 흐름을 거쳤기 때문에 여기까지 왔는지가 더 중요하다."

각 콘텐츠에서 **결론에 도달하기 전의 사고 과정**을 명시적으로 서술해야 한다. 면접관이 읽었을 때 자연스럽게 떠오르는 질문들을 글 안에서 선제적으로 답해야 한다.

**면접관이 물을 법한 질문 → 글 안에 답이 있어야 하는 것:**

| 콘텐츠 | 예상 면접 질문 | 글에 있어야 할 답 |
|--------|---------------|-------------------|
| `cesium-adoption.md` | "왜 Cesium이고 Mapbox GL JS나 deck.gl은 안 됐나?" | 3가지 선택지 비교에 "왜 이걸 선택 안 했는지"가 명시 |
| `aws-lambda-ec2.md` | "왜 서버리스(Fargate)를 안 쓰고 EC2를 직접 관리했나?" | 팀 규모가 작아서 DevOps 인력이 없었지만, 고사양 EC2를 짧게 돌리는 것이 총 비용에서 유리했다는 근거 |
| `ktx2-basis-universal.md` | "ETC1S 대신 UASTC를 선택했으면 어떻게 됐을까?" | GPU 메모리 2~4배 증가를 감수해야 했고, 드론 항공 이미지에서는 품질 차이가 체감되지 않았다는 판단 |
| `fbxloader-memory-leak.md` | "dispose()로 3배 줄였는데 왜 그게 충분하지 않았나?" | Retained Size는 줄었지만 메모리 변동폭 자체는 개선되지 않았다. 라이브러리의 구조적 한계 |
| `coordinate-transform.md` | "proj4js 말고 다른 좌표 변환 라이브러리는 검토했나?" | (현재 빠져 있음 → 추가 필요) |
| `ifc-3dtiles-pipeline.md` | "5단계 중 가장 시간이 오래 걸린 단계는?" | (현재 빠져 있음 → 단계별 처리 시간 추가 필요) |

**작성 원칙**: 글을 다 쓴 후 "이 글을 읽은 면접관이 어떤 질문을 할까?"를 3개 이상 떠올려 보고, 그 답이 글 안에 없으면 추가한다.

### 3-4. 글 간의 내러티브 연결: 하나의 이야기

> "자기소개에 맞추다 보니까 조금 더 기술적인 임팩트 있는 것들을 쓰게 되더라고. 자기소개에서부터 컨셉이 정해졌고, 이 사람의 인격을 가졌다면 어떤 식으로 이력서가 정해져야 될지 방향이 잡힌 거네요."

이 블로그의 21개(+2개 신규) 글은 **하나의 이야기의 챕터**로 읽혀야 한다. 각 글은 독립적으로도 읽을 수 있지만, 순서대로 읽으면 하나의 기술 여정이 된다.

```

  coordinate-transform → geoid-correction → meridian-convergence → model-matrix

제4막: 최적화 — 브라우저에서 버틸 수 있게 만들다
  gpu-texture-formats → ktx2-basis-universal → draco-compression → lod → spatial-partitioning

제5막: 자동화 — 수작업을 파이프라인으로 전환하다
  ifc-3dtiles-pipeline → aws-lambda-ec2

제6막: 기능 — 사용자가 실제로 쓸 수 있는 도구를 만들다
  measurement-tools → area-calculation → cesium-mouse-events → optimistic-update

에필로그: 프론트엔드 자체의 성능도 잡다
  react-compiler-optimization
```

**각 글에서 이 연결을 드러내는 방법:**
- 글의 도입부나 마지막에 "이 문제는 [이전 글]에서 발견됐다" 또는 "이 결과가 [다음 글]의 출발점이 됐다"를 한 줄 넣는다
- 독자가 그래프에서 다음 노드로 이동하고 싶게 만드는 장치

### 3-5. 경험을 콘텐츠로 승화시키는 기술

> "별로여 보이는 문제 해결일지라도 스토리나 로직들이 명확하고 이 사람의 엔지니어링 쪽 사고를 드러내 줄 수 있다면 전 그것도 되게 매력적인 거 같아요."

모든 경험이 "GPU 메모리 73% 절감" 같은 화려한 수치를 갖고 있지는 않다. 중요한 건 수치의 크기가 아니라 **사고 과정의 깊이**다.

#### "별로여 보이는" 경험을 매력적으로 만드는 방법

**사례: `meridian-convergence.md` (자오선 수차 보정)**

이 경험은 겉보기에 "그냥 0.5도 회전 보정"이다. 화려하지 않다. 하지만 아래와 같이 쓰면 달라진다:

```
[별로여 보이는 버전]
자오선 수차를 적용하여 모델의 회전 오차를 보정했다.

[매력적인 버전]
건물 BIM 모델을 Cesium에 올렸을 때, 건물 끝단이 실제 위치와 수 미터 어긋나 있었다.
좌표 변환은 정확했고, 지오이드고 보정도 했다. 나머지 오차의 원인이 뭘까?

원인은 Grid North와 True North의 차이였다.
한국 중부 좌표계(EPSG:5186)의 중앙 자오선은 127°E인데,
부산(129°E)에서는 이 차이가 0.5~1도의 회전 오차를 만든다.
수백 미터짜리 건물에서 0.5도는 끝단에서 수 미터의 위치 차이다.

이걸 발견하기까지 좌표 변환 코드를 세 번 다시 검토했다.
좌표 변환 자체에는 문제가 없었기 때문에 원인 파악이 더 어려웠다.
```

**차이점**: "0.5도 보정"이라는 사실은 같다. 하지만 두 번째 버전에서는:
- 이걸 왜 찾아야 했는지 맥락이 있다 (건물 끝단이 어긋남)
- 원인을 찾는 과정의 어려움이 있다 (좌표 변환 자체에는 문제가 없었으므로)
- 수치가 실제 영향으로 번역되어 있다 (0.5도 = 수 미터)

#### 경험을 승화시키는 3단계

**1단계: 사건의 핵심을 찾아라**
"내가 이 작업을 하면서 가장 어려웠던 순간은 언제인가?" — 그 순간이 글의 도입부가 된다.
- 고객사 시연 중 크래시 → `fbxloader-memory-leak.md`의 도입부
- 모델이 화면에 안 보임 → `geoid-correction.md`의 도입부
- 코드에는 문제가 없는데 결과가 틀림 → `meridian-convergence.md`의 도입부

**2단계: "왜 쉽게 해결하지 못했는가"를 설명하라**
쉬운 문제를 풀었다면 글을 쓸 이유가 없다. 독자가 알고 싶은 건:
- 처음에 어디를 잘못 봤는지 (잘못된 가설)
- 어떤 시행착오를 거쳤는지 (탐색 과정)
- 결국 어떻게 원인을 특정했는지 (결정적 단서)

이것은 "삽질했다"와 다르다. 시행착오의 과정 자체가 엔지니어링 사고를 보여주는 증거다.

**3단계: "이걸 통해 무엇을 알게 됐는가"를 한 줄로 정리하라**
- `fbxloader-memory-leak.md`: "라이브러리의 구조적 한계는 사용법 개선으로 해결되지 않는다"
- `coordinate-transform.md`: "좌표값이 맞아도 좌표계가 다르면 결과는 완전히 틀리다"
- `meridian-convergence.md`: "좌표 변환이 정확해도, 방위각의 미세한 차이가 대형 구조물에서는 눈에 보이는 오차가 된다"

이 한 줄이 글의 **결론이자 독자가 기억할 핵심**이 된다.

#### 전체 문단 리라이팅 사례: `aws-lambda-ec2.md` "대안 검토" 섹션

현재:
```
## 대안 검토: 왜 EC2인가

AWS에서 장시간 배치 작업을 처리하는 서비스는 여러 가지가 있다:

- **AWS Batch**: 배치 작업 관리 서비스. 설정이 복잡하고 관리 오버헤드가 있다.
- **ECS Fargate**: 서버리스 컨테이너. 편리하지만 비용이 EC2보다 높다.
- **EC2**: 직접 인스턴스를 관리해야 하지만, 비용이 가장 저렴하다.

최종적으로 EC2가 가장 비용 효율적이었다.
```

문제: 각 대안의 설명이 1줄로 너무 짧다. "왜 이걸 선택 안 했는지"의 근거가 빈약하다. 면접관이 "왜 Fargate 안 썼어요?"라고 물으면 "비싸서요"밖에 답할 수 없다.

개선안:
```
## 대안 검토: 왜 EC2인가

Lambda에서 직접 변환이 불가능하다면, 별도의 컴퓨팅 리소스가 필요하다.
AWS에서 이 역할을 할 수 있는 서비스 세 가지를 검토했다.

| 서비스 | 장점 | 단점 | 우리 상황에서의 판단 |
|--------|------|------|---------------------|
| AWS Batch | 다중 인스턴스 자동 오케스트레이션 | 초기 설정 복잡, Job Queue/Compute Environment 구성 필요 | DevOps 인력이 없는 소규모 팀에서 설정·유지 부담이 과도 |
| ECS Fargate | 서버리스, 인스턴스 관리 불필요 | 동일 사양 대비 EC2보다 비용 20~30% 높음 | 변환 빈도가 월 수십 회 수준이라 비용 차이가 누적됨 |
| EC2 | 비용 최저, 사양 자유 선택 | 인스턴스 시작/종료를 직접 관리해야 함 | Lambda가 트리거하므로 관리 부담이 거의 없음 |

EC2를 선택한 핵심 근거는 **"높은 사양을 짧게 돌리는 것이 낮은 사양을 오래 돌리는 것보다 총 비용이 적다"**는 전략이다.
32 vCPU 인스턴스를 사용하면 CPU 코어 수만큼 병렬 처리가 가능하여,
3,561개 타일을 약 10분 만에 변환할 수 있었다.

다만, 변환 대상이 10만 개 이상으로 증가하면 단일 인스턴스의 한계에 부딪힌다.
그 시점에서는 AWS Batch로의 전환이 필요하다.
```

**개선 포인트 정리:**
- 각 대안의 "왜 안 됐는지"에 팀 상황(DevOps 인력 없음, 소규모 팀) 맥락 추가
- 비용 비교에 구체적 수치(20~30% 높음) 추가
- 핵심 전략("높은 사양을 짧게")을 볼드로 강조
- 마지막에 한계와 다음 단계까지 언급 → 확장 사고

### 3-6. 콘텐츠 노드의 표준 구조

에이전트가 노드를 작성하거나 수정할 때 따라야 하는 **섹션 순서와 역할**:

```markdown
# {제목} ← 문제 또는 결과가 드러나는 제목 (5-2절 참고)

## 한 줄 요약 ← 3초 안에 "읽을지 말지" 결정하게 만든다 (4-5절 참고)

## {왜 이것이 필요했는가} ← Problem. 실무 에피소드로 시작 (4-6절 도입부 패턴)
  - 상황 → 긴장 → 질문 → 결과 미리보기
  - 이 노드의 전제가 되는 다른 노드 언급 ("이 문제는 [노드]에서 시작됐다")

## {대안 검토 / 원인 분석} ← Analyze. 이 글의 핵심 가치 (3-7절 참고)
  - 최소 2개 이상의 선택지 비교
  - 각 선택지의 트레이드오프 (표 또는 목록)
  - "우리 상황에서는 X가 맞았다"는 근거 + 팀/비용/기술 맥락
  - 선택하지 않은 대안이 더 나은 상황도 언급

## {구현 / 적용 과정} ← 핵심 기술 내용. 코드·다이어그램·수식 포함
  - 코드 블록 위에 한 줄 설명 필수
  - 전문 용어 첫 등장 시 괄호 설명

## {결과} ← Result. 수치 + 측정 방법
  - Before/After 수치 (절대값 + 비율)
  - 측정 도구/조건 명시 ("Chrome DevTools Memory 탭에서 측정")
  - 한계점이나 남은 과제도 솔직하게

## {더 큰 규모에서는? / 다른 환경에서는?} ← 확장 사고 (선택적, 핵심 노드에만)
  - "만약 ~였다면" 가정을 통해 독자의 상황에 대입 가능하게

(마지막 줄) 독자가 즉시 실행할 수 있는 것 한 줄 ← 4-7절 참고
```

**섹션 제목은 고정 아님.** 위 구조의 **역할**을 유지하되, 제목은 해당 노드의 맥락에 맞게 변경한다.
예: "대안 검토" 대신 "왜 Lambda에서 직접 변환하지 않는가", "왜 EC2인가" 등.

### 3-7. Analyze 섹션 작성 가이드

PAR 중 **A가 가장 어렵고, 가장 가치 있다**. 에이전트가 A 섹션을 쓸 때 반드시 포함해야 하는 요소:

#### 필수 요소 4가지

**1. 선택지 나열** — 최소 2개, 이상적으로 3개
```
검토한 대안:
(1) AWS Batch — 다중 인스턴스 오케스트레이션
(2) ECS Fargate — 서버리스 컨테이너
(3) EC2 — 직접 인스턴스 관리
```

**2. 각 선택지의 트레이드오프** — 반드시 표 또는 구조화된 비교
장점만 나열하지 않는다. **"왜 이걸 안 골랐는지"**가 핵심이다.
```
| 서비스 | 장점 | 단점 | 우리 상황에서의 판단 |
```
"우리 상황에서의 판단" 열이 가장 중요하다. 여기에 팀 규모, 예산, 기존 인프라 등 **맥락**이 들어간다.

**3. 최종 선택의 근거** — 한 문장으로 압축
```
"DevOps 인력이 없는 2인 팀에서, 고사양 EC2를 짧게 돌리는 것이 총 비용에서 가장 유리했다."
```
이 한 문장에 (1) 팀 상황 (2) 선택 (3) 근거가 모두 있어야 한다.

**4. 선택하지 않은 대안이 더 나은 상황** — 독자를 위한 배려
```
"변환 대상이 10만 개 이상이라면 AWS Batch가 더 적합하다."
```
이것이 있으면 독자는 "이 사람은 자기 상황만 아는 게 아니라 다른 상황도 고려한다"고 느낀다.

#### Analyze가 불가능한 콘텐츠의 처리

일부 노드는 "선택"이 아니라 "발견"이다 (예: `meridian-convergence.md`).
이 경우 A 섹션은 **"원인 추적 과정"**으로 대체한다:

```
1. 처음에 의심한 원인 (잘못된 가설)
2. 그 가설을 배제한 근거
3. 실제 원인을 특정한 결정적 단서
4. 이 경험에서 얻은 교훈
```

### 3-8. Gold Standard: `fbxloader-memory-leak.md`를 참고 모델로

현재 노드 중 PAR 구조와 글의 품질이 가장 높은 글은 `fbxloader-memory-leak.md`다.
에이전트가 다른 노드를 수정하기 전에 이 글을 반드시 먼저 읽고 톤·구조·깊이를 참고해야 한다.

**이 글이 잘 된 이유:**

| 요소 | 이 글에서의 구현 | 다른 노드에 적용할 때 |
|------|-----------------|---------------------|
| **P (도입부)** | "고객사 시연 중 크래시" — 구체적 상황, 비즈니스 임팩트 | 모든 노드에 이 수준의 실무 에피소드 필요 |
| **A (원인 추적)** | Chrome DevTools 프로파일링 → Shallow vs Retained Size 비교 → FBXLoader 특정 | 분석 과정을 단계별로 보여줌. "처음엔 X를 의심 → 아니었음 → Y가 원인" |
| **R (결과와 한계)** | Retained Size 3배 감소. 하지만 "근본 해결은 아니었다" → Cesium 전환 결정 | 결과만이 아니라 **한계까지 솔직하게** 쓴 점이 신뢰를 준다 |
| **내러티브** | 다음 노드(`cesium-adoption`)로 자연스럽게 이어진다 | 글 마지막에 다음 노드 연결 필요 |
| **톤** | 전문적이되 과시하지 않음. 시행착오를 솔직하게 서술 | "삽질했다" 대신 "이 접근으로는 한계가 있었다" |

**에이전트에게 주는 지시**: 노드를 수정하거나 새로 작성할 때, `fbxloader-memory-leak.md`의 도입부 구조, 분석 깊이, 결과의 솔직함을 기준으로 삼는다. 이 글보다 얕으면 보강이 필요하다.

### 3-9. 성과 표현의 Before/After

| 나쁜 표현 (추상적) | 좋은 표현 (구체적) | 무엇이 달라졌는가 |
|-------------------|-------------------|-------------------|
| "할 수 있게 됨" | "GPU 메모리 75MB → 20MB (73% 절감)" | 추상 → 수치 |
| "하는 계기가 되었음" | "고객사 시연 중 OOM 크래시 발생 → 원인 규명 후 Retained Size 3배 감소" | 감상 → 사건+결과 |
| "성능이 개선되었다" | "초기 로드 시간 28초 → 20초 (28% 단축), Chrome DevTools Performance 탭에서 측정" | 모호 → 수치+측정 방법 |
| "안정적으로 운영 중" | "메모리 변동폭 8.0~8.9GB → 6.9~7.2GB (66% 안정화)" | 주관 → 범위+변화율 |
| "많은 기능을 추가했다" | "Potree 28개 기능 → Cesium 43개 기능 (+15개 신규)" | "많은" → 정확한 수 |
| "빠르게 처리된다" | "3,561개 타일을 8 워커로 병렬 처리하여 약 10분에 완료" | "빠르게" → 처리량+시간 |

## 4. 글쓰기 톤 & 스타일

이 블로그의 문체: **"시니어 개발자가 후배에게 설명해 주는 톤"**. 쉽되 가볍지 않고, 전문적이되 딱딱하지 않다.

### 4-1. 원칙 1: 구체적으로 써라

"무슨 일이 일어났는지"를 기술 용어로 정확하게 묘사한다. 구어체 과장은 가볍게 보이고, 모호한 서술은 이해를 방해한다.

| 나쁜 문장 (모호/구어체) | 좋은 문장 (구체적/전문적) | 왜 나쁜가 |
|------------------------|--------------------------|-----------|
| "서비스가 터졌다" | "브라우저 탭의 메모리 사용량이 한계를 초과하여 OOM 크래시가 발생했다" | "터졌다"는 원인이 빠져 있고 가벼움 |

| "문제가 생겼다" | "페이지를 벗어나도 메모리가 해제되지 않고 누적되면서 점진적 성능 저하가 발생했다" | 어떤 문제인지 알 수 없음 |

**핵심 규칙: 형용사/부사 대신 숫자와 기술 용어를 써라.**

### 4-2. 원칙 2: 쉽게 써라

전문적으로 쓰는 것과 어렵게 쓰는 것은 다르다. 전문 용어는 사용하되, 처음 등장할 때 괄호로 한 줄 설명을 붙인다.

| 나쁜 문장 | 좋은 문장 |
|-----------|-----------|
| "Retained Size가 비정상적으로 컸다" | "Retained Size(이 객체가 참조하는 전체 객체 트리의 크기)가 비정상적으로 컸다" |
| "ETC1S 코덱을 사용했다" | "ETC1S 코덱(품질보다 압축률을 우선하는 모드)을 사용했다" |
| "frustum culling을 적용했다" | "frustum culling(카메라 시야 밖의 객체를 렌더링에서 제외하는 기법)을 적용했다" |

규칙:
- 전문 용어 첫 등장 시 괄호로 1줄 설명 → 이후에는 용어만 사용
- 약어(OOM, LOD, GPU 등)도 첫 등장 시 풀어쓰기
- 기본 용어(HTML, CSS, API 등)는 대상 독자가 개발자이므로 풀지 않음

### 4-5. 한 줄 요약 작성법

"한 줄 요약"은 독자가 이 글을 읽을지 말지 3초 안에 결정하게 만든다.

- **약한 한 줄 요약**: 정의를 설명한다 ("A는 B를 하는 기술이다")
- **강한 한 줄 요약**: 문제 → 해결 → 결과를 한 문장에 압축한다

실제 콘텐츠 Before/After:

`spatial-partitioning.md` 현재:
> "대용량 3D 데이터를 웹에서 효율적으로 렌더링하려면, 데이터를 공간적으로 분할하여 필요한 부분만 로드해야 한다. Quadtree와 AABB는 이 분할에 사용되는 대표적인 두 가지 기법이다."

문제: "대표적인 기법이다"라는 사전적 정의. 이 글을 왜 읽어야 하는지 동기가 없다.

개선안:
> "수천 개의 건축 요소를 하나의 파일로 웹에 올리면, 카메라에 보이지 않는 부분까지 GPU에 올라가면서 메모리가 낭비된다. Quadtree로 공간을 분할하고 AABB로 시야 판정을 하면, 필요한 타일만 선택적으로 로드할 수 있다."

`3d-tiles-spec.md` 현재:
> "OGC가 정의한 대용량 3D 데이터 스트리밍 포맷. 타일 기반 공간 분할, LOD, 비동기 로딩을 통해 브라우저에서도 도시 규모 모델을 렌더링할 수 있게 한다."

문제: OGC라는 약어로 시작. "~할 수 있게 한다"로 끝남.

개선안:
> "수천 개의 3D 타일을 브라우저에서 한 번에 로드하면 메모리 사용량이 급증하여 탭이 크래시된다. 3D Tiles는 카메라가 보는 영역의 타일만, 필요한 해상도로 로드하여 이 문제를 해결한다. 실제 운영 환경에서 3,561개 타일을 이 방식으로 안정적으로 서비스했다."

`cesium-adoption.md` 현재:
> "메모리 누수, 고객사의 Mesh 시각화 요구, 기존 뷰어의 구조적 한계가 겹치면서, 1인 마이그레이션이라는 리소스 제약 속에서 비용·커스텀 자유도·데이터 보안·대용량 Mesh 지원을 기준으로 세 가지 선택지를 비교한 끝에 "자체 서버 Builder + Cesium.js"를 선택했다."

문제: 한 문장에 정보가 과적되어 있다. 읽다가 숨이 차다.

개선안:
> "FBXLoader의 구조적 메모리 누수, Mesh 시각화 불가, 2D/3D 지도 부재 — 세 가지 한계가 동시에 드러났다. 비용·커스텀 자유도·데이터 보안을 기준으로 세 선택지를 비교한 끝에 Cesium.js 자체 구축을 결정했다."

### 4-6. 도입부 패턴: 상황 → 긴장 → 질문 → 결과

채용 담당자는 글을 **스캔**한다. 첫 2~3문단에서 "이 사람은 깊이가 있다"는 인상을 줘야 한다.

```
[상황]  고객사 시연 중, 3D 모델 페이지에서 반복적으로 흰 화면이 나타났다.
[긴장]  브라우저 탭이 자동 리로드되면서 시연이 중단됐다.
        Chrome의 OOM(Out of Memory) 보호 메커니즘이 작동한 것이었다.
[질문]  페이지를 벗어났는데 왜 메모리가 해제되지 않았을까?
[결과]  Retained Size 분석을 통해 FBXLoader의 참조 체인이 끊기지 않는 구조적 문제를 발견했고,
        dispose()로 Retained Size를 3배 감소시켰지만 근본 해결은 Cesium 전환이었다.
```

이 패턴은 `fbxloader-memory-leak.md`에 이미 적용되어 있다. 다른 글에도 동일하게 적용한다.

**도입부가 약한 노드별 구체적 리라이팅:**

`coordinate-transform.md` 현재 도입:
> "로컬 평면 좌표를 Cesium 내부 좌표로 변환하는 과정에서 float64 정밀도 유지가 핵심이다."

문제: 기술 설명으로 시작. "왜 이게 문제인데?"가 없다.

개선안:
```
[상황]  BIM 모델을 Cesium에 올렸는데, 모델이 서울이 아니라 의도한 위치에서
        수백 km 떨어진 지점에 렌더링됐다.
[긴장]  좌표값 자체는 정확했다. 좌표계(EPSG:5186)와 Cesium 내부 좌표계(ECEF)의
        원점과 축이 다르다는 사실을 모른 채 변환 없이 그대로 입력한 것이 원인이었다.
[질문]  EPSG:5186 → WGS84 → ECEF, 이 세 단계 변환을 어떻게 정확하게 수행할 수 있을까?
[결과]  proj4js로 변환 파이프라인을 구축하고, float64 정밀도 유지로
        오차를 1~5cm 이내로 줄였다.
```

`model-matrix.md` 현재 도입:
> "좌표 변환·지오이드고·자오선 수차 보정값을 4×4 행렬로 합성하여 모델 배치."

문제: 수학 요약이지 도입부가 아니다.

개선안:
```
[상황]  좌표 변환을 거쳐 Cesium에 올린 건물 모델이 90도 옆으로 누워 있었다.
[긴장]  좌표는 맞았는데 방향이 틀렸다. BIM 도구는 Y축이 위를 가리키고(Y-up),
        Cesium은 Z축이 위를 가리킨다(Z-up). 축 관례의 차이였다.
[질문]  위치(translation), 회전(rotation), 스케일(scale)을 하나의 행렬로

[결과]  Scale → Rotation → Translation 순서로 4×4 행렬을 합성하여
```

`geoid-correction.md` 현재 도입:
> "BIM의 정표고와 Cesium terrain의 타원체고 차이를 20~30m 보정."

### 4-7. 글의 마무리: "이 글에서 얻어갈 것"

유튜브 스크립트에서 마지막에 강조한 것:
> "오늘부터 실행할 수 있는 것. 첫째, 이력서를 열어서 Problem 없이 Action만 서술된 것을 찾아라. 둘째, 가장 잘했다고 생각하는 프로젝트를 골라 PAR 초안을 작성하라. 셋째, 수치가 없는 문장마다 숫자를 채워라."

각 콘텐츠의 마지막에도 독자가 **즉시 실행할 수 있는 것**을 한 줄 넣는다:

- `fbxloader-memory-leak.md`: "메모리 누수가 의심된다면, Chrome DevTools Memory 탭에서 페이지 이동 전후의 Retained Size를 비교해 보라."
- `ktx2-basis-universal.md`: "현재 프로젝트의 텍스처를 `gltf-transform inspect`로 분석해 보라. RGBA 비압축 텍스처가 있다면 KTX2 적용 대상이다."
- `aws-lambda-ec2.md`: "S3 업로드 이벤트를 Lambda로 트리거하는 것은 AWS 콘솔에서 10분이면 설정할 수 있다. 반복 작업을 자동화할 기회가 있다면 시도해 보라."

### 4-8. 문단/코드 블록 규칙

- 한 문단 = 3~5줄. 문단 첫 문장이 핵심(topic sentence)
- 비교 3개 이상이면 표 또는 목록으로 전환
- 코드 블록 바로 위에 "이 코드가 무엇을 하는지" 한 줄 설명 필수

### 4-9. 자가 점검 체크리스트

**가독성 점검:**
1. 소리 내어 읽었을 때 어색한 부분이 없는가?
2. 형용사/부사를 숫자로 대체할 수 있는 곳이 있는가? — "크게 개선" → "73% 감소"
3. 전문 용어 첫 등장에 괄호 설명이 있는가?
4. 한 문장이 60자를 넘는 곳이 있는가? — 마침표로 끊는다
5. 금지 표현(4-4)이 있는가?
6. 이 글을 처음 보는 주니어 개발자가 따라올 수 있는가?

**PAR 점검:**
7. Problem: "이걸 왜 해야 했는가"가 첫 3줄 안에 있는가?
8. Analyze: 최소 2개 이상의 대안을 비교하고 "왜 이걸 선택했는지" 근거가 있는가?
9. Result: 수치가 있는가? 수치 옆에 측정 방법이 있는가?
10. 이 글을 읽은 면접관이 물을 법한 질문 3개를 떠올리고, 그 답이 글 안에 있는가?

**내러티브 점검:**
11. 이 글의 도입부에 실무 에피소드(상황→긴장→질문→결과)가 있는가?
12. 다른 노드와의 연결("이 문제는 [노드]에서 시작됐다")이 한 줄이라도 있는가?
13. 글 마지막에 독자가 즉시 실행할 수 있는 것이 있는가?
14. 이 글이 블로그 페르소나("3D 렌더링 성능에 집착하는 엔지니어")와 일관되는가?

## 5. 그래프 구조 재설계

### 5-1. 클러스터 재편: 9개 → 6개

현재 9개 클러스터 중 `format`(1개 노드)과 `decision`(1개 노드) 같은 1-노드 클러스터가 존재한다. 시각적으로 고립되고 그래프 탐색 시 자연스러운 흐름을 방해한다.

**재편 원칙**: 독자가 블로그를 탐색할 때의 **사고 흐름**을 따른다.
"문제 발견 → 데이터 이해 → 좌표 배치 → 성능 최적화 → 파이프라인 자동화 → 기능 구현"

| 새 클러스터 ID | 한글명 | 포함 노드 | 변경 사항 |
|---------------|--------|-----------|-----------|
| `discovery` | 문제 발견과 결정 | fbxloader-memory-leak, chrome-devtools-memory, cesium-adoption, **potree-cesium-migration (신규)** | `problem` + `decision` 통합. 이 블로그 전체의 출발점이 되는 클러스터 |
| `data` | 3D 데이터 기초 | 3d-tiles-spec, mesh-bim-pointcloud | `format` + `graphics` 일부 통합. "어떤 데이터를 다루는가" |
| `coordinate` | 좌표와 배치 | coordinate-transform, geoid-correction, meridian-convergence, model-matrix, area-calculation | `geodesy` + `implementation` 일부. model-matrix는 좌표 배치의 최종 단계이므로 여기로 이동 |
| `performance` | 성능 최적화 | gpu-texture-formats, ktx2-basis-universal, draco-compression, lod, spatial-partitioning, **react-compiler-optimization (신규)** | `optimization` + `graphics` 일부 통합. "어떻게 빠르게 만드는가" |
| `pipeline` | 데이터 파이프라인 | aws-lambda-ec2 **(docker-container 병합)**, ifc-3dtiles-pipeline | `infrastructure` + `implementation` 일부 통합. "데이터 변환을 어떻게 자동화하는가" |
| `feature` | 기능 구현 | measurement-tools, cesium-mouse-events, optimistic-update | `frontend` + `implementation` 일부 통합. "사용자에게 어떤 기능을 제공하는가" |

**제거되는 클러스터**: `format`, `decision`, `problem`, `graphics`, `optimization`, `infrastructure`, `frontend`, `implementation`
**신설되는 클러스터**: `discovery`, `data`, `performance`, `pipeline`, `feature`
**유지되는 클러스터**: `coordinate` (기존 `geodesy`에서 이름 변경)

### 5-2. 노드 변경: 병합, 분리, 신규, 제목 변경

#### 병합: `docker-container` → `aws-lambda-ec2`에 통합

이유:
- `docker-container` 단독으로는 연결이 2개뿐 (aws-lambda-ec2, ktx2-basis-universal)
- Docker 이미지 선택(node:slim vs full vs alpine)은 EC2 파이프라인의 **구현 세부사항**
- 독립 노드로서의 깊이가 부족 (97줄, 다른 노드 대비 얕음)

실행:
- `aws-lambda-ec2.md`에 "Docker 환경 구성" 섹션 추가
- node:slim 선택 이유, 이미지 크기 비교(~200MB vs ~900MB), Dockerfile 구조 포함
- `docker-container.md` 파일 삭제
- 관련 노드의 `relatedConcepts`에서 `docker-container` 참조 제거

#### 신규 노드 1: `react-compiler-optimization`

이유:

- 3D/GIS가 아닌 일반 React 개발자도 관심을 가질 수 있어 독자 확장 효과가 큼

### 6-2. 전체 노드 S++ 리라이팅 지시

**모든 노드가 전면 리라이팅 대상이다.** Tier 구분 없이, 각 노드를 S++ 기준에 도달시키기 위해 필요한 구체적 작업을 명시한다. 에이전트는 아래 지시사항에 명시되지 않은 노드라도 S++ 기준표(문서 상단)에 맞춰 자체 판단으로 리라이팅한다.

#### 모든 노드 공통 작업 (노드별 지시 전에 반드시 수행)

1. **도입부 리라이팅**: 정의/설명으로 시작하는 도입부를 상황→긴장→질문→결과 패턴으로 교체 (4-6절)
2. **한 줄 요약 교체**: "~이다" 정의형 → "~했더니 ~됐다" 결과형 (4-5절)
3. **Analyze 섹션 강화**: 대안 비교가 없거나 약한 경우, 3-7절 가이드에 따라 최소 2개 대안 비교 + 트레이드오프 표 + 선택 근거 추가
4. **결과 수치화**: "개선됨", "해결됨" 등 모호한 표현을 모두 Before/After 수치로 교체. 측정 도구/조건 명시
5. **외부 권위 인용**: 6-3절 인용 가이드에 해당 노드가 있으면 반드시 인용 삽입
6. **내러티브 연결**: 3-4절 내러티브 아크에서 이 노드의 위치를 확인하고, 이전/다음 노드와의 연결 문장 1줄 추가
7. **면접 질문 선제 답변**: 이 글을 읽은 면접관이 물을 법한 질문 3개를 떠올리고, 그 답이 글 안에 없으면 추가 (3-3절)
8. **독자 액션 마무리**: 글 마지막에 독자가 즉시 실행할 수 있는 것 1줄 추가 (4-7절)
9. **금지 표현 제거**: 4-4절 목록의 표현을 전수 검색하여 교체
10. **기밀 점검**: 2장 기준에 따라 최종 점검
