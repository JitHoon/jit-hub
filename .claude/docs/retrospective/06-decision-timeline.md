## 개요

프로젝트 방향을 결정지은 주요 기술 결정들을 시간순으로 분석한다. 결정의 트리거, 대안, 선택 근거, 결과를 기록하여 다음 프로젝트에서 유사한 갈림길에서 더 빠르게 판단할 수 있게 한다.

## 주요 의사결정 타임라인

### Decision 1: 프로젝트 초기 기술 스택 선정 (Day 1)

| 항목 | 내용 |
|------|------|
| 트리거 | 3D GIS 지식 포트폴리오를 어떤 스택으로 구현할 것인가 |
| 대안 | (A) Gatsby + MDX (B) Astro + three.js (C) Next.js + react-force-graph |
| 선택 | (C) Next.js 16 + react-force-graph-3d |
| 근거 | App Router의 SSG/SSR 하이브리드, Vercel 배포 최적화, React 생태계 활용 |
| 결과 | ✅ 올바른 선택 — SEO + 인터랙티브 시각화 양립 |

### Decision 2: 콘텐츠 파이프라인 Velite → 커스텀 전환 (Day 2~3)

| 항목 | 내용 |
|------|------|
| 트리거 | Velite 0.x의 불안정성, 단일 유지보수자 리스크, Turbopack 비호환 |
| 대안 | (A) Velite 유지 (B) Contentlayer (C) gray-matter + Zod 커스텀 |
| 선택 | (C) 커스텀 파이프라인 |
| 근거 | Contentlayer도 유지보수 중단, 커스텀 파이프라인은 완전한 제어권 확보 |
| 결과 | ✅ 올바른 선택 — Zod 검증 + 빌드 타임 무결성 검사 구현 |
| 비용 | 약 1일 추가 개발 시간 |

### Decision 3: 클러스터 7개 → 9개 리밸런싱 (Day 3)

| 항목 | 내용 |
|------|------|
| 트리거 | "지식" 클러스터에 38% 노드 집중 (불균형) |
| 대안 | (A) 7개 유지 (B) 8개 (C) 9개 |
| 선택 | (C) 9개 클러스터 |
| 근거 | 시각적 균형, 그래프 탐색 시 인지 부하 감소 |
| 결과 | ✅ 균형 잡힌 분포 달성 |

### Decision 4: 단일 페이지 분할 뷰 결정 (Day 3~4)

| 항목 | 내용 |
|------|------|
| 트리거 | 노드 클릭 시 별도 상세 페이지로 이동할 것인가 |
| 대안 | (A) 별도 상세 페이지 (B) 분할 뷰 (C) 모달 |
| 선택 | (B) 그래프(좌) + 본문(우) 분할 뷰 |
| 근거 | 그래프 맥락 유지, 탐색 연속성, 포트폴리오 특성(한 페이지에서 모든 것) |
| 결과 | ✅ UX 차별화 포인트 |
| 보완 | SEO용 /nodes/[slug] 정적 페이지 별도 생성 |

### Decision 5: 2-tier 관계 모델 채택 (Day 3)

| 항목 | 내용 |
|------|------|
| 트리거 | 노드 간 관계를 어떻게 표현할 것인가 |
| 대안 | (A) 단일 관계 타입 (B) 다중 관계 타입 (C) 2-tier (데이터 + 시각) |
| 선택 | (C) 2-tier 관계 모델 |
| 근거 | 데이터 레벨에서는 자유도 높게, 시각 레벨에서는 4~5 카테고리로 제한 |
| 결과 | ✅ 시각적 혼잡 방지 + 데이터 유연성 양립 |

### Decision 6: v1 → v2 블루프린트 전환 (Day 3~4)

| 항목 | 내용 |
|------|------|
| 트리거 | 초기 설계(v1)에서 10개 주요 변경사항 발생 |
| 대안 | (A) v1 패치 (B) v2 문서 작성 |
| 선택 | (B) project-blueprint-v2.md 작성 (259줄) |
| 근거 | 변경사항이 누적되어 v1 패치로는 일관성 유지 불가 |
| 결과 | ✅ 명확한 의사결정 기록 — 이후 모든 작업의 나침반 역할 |

### Decision 7: stage 필드 제거 (Day 4)

| 항목 | 내용 |
|------|------|
| 트리거 | 21개 노드 모두 에버그린 콘텐츠, stage 구분 불필요 |
| 대안 | (A) stage 유지 (B) stage 제거 |
| 선택 | (B) 제거 |
| 근거 | YAGNI 원칙 — 현재 불필요한 필드는 제거 |
| 결과 | ✅ 스키마 단순화, 프론트매터 작성 부담 감소 |

### Decision 8: Hub-centric 그래프 탐색 전략 (Day 4~5)

| 항목 | 내용 |
|------|------|
| 트리거 | 21개 노드를 한 번에 표시하면 스파게티 |
| 대안 | (A) 전체 표시 (B) 클러스터별 표시 (C) 허브 5개 + 호버 시 리프 표시 |
| 선택 | (C) Hub-centric |
| 근거 | 점진적 탐색, 시각적 혼잡 방지, 확장성 (노드 증가 시에도 유지) |
| 결과 | ✅ 직관적 탐색 경험 |

## 의사결정 패턴 분석

### 의사결정 속도
- 대부분의 결정이 Day 1~4에 집중 (전체 기간의 57%)
- Day 5 이후는 구현과 폴리시에 집중
- **교훈**: 첫 주 초반에 핵심 결정을 빠르게 내리고 실행에 집중하는 패턴이 효과적

### 의사결정 품질
- 8개 주요 결정 중 번복된 것: 0개
- v1→v2 전환은 "번복"이 아닌 "진화" (초기 가설 → 실험 → 개선)
- **교훈**: blueprint 문서로 결정 근거를 명시하면 번복 충동을 줄일 수 있음

### 의사결정 도구
- project-blueprint-v2.md가 핵심 의사결정 저장소 역할
- CLAUDE.md의 절대 규칙이 일상적 결정을 자동화
- **교훈**: 의사결정을 문서화하면 AI 협업 시에도 일관성 유지 가능
