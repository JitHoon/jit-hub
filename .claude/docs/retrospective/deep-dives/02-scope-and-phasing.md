## 배경

대시보드 리뷰 포인트 #1 + #7:

- #1: "하네스 엔지니어링을 통해 7일간 만든 프로젝트 규모가 적절했는지 궁금하다."
- #7: "이런 개발 사이클 경험이 처음이라서 Phase별 진행이 뒤죽박죽이었다. 이 프로젝트를 처음부터 다시 구축한다면 가장 이상적이고 정석적인 소프트웨어 설계 및 구현 과정이 궁금하다."

7일간 267 커밋, 34 PR, 4,644 LOC를 달성했다. 이 규모가 적절했는지, Phase 순서를 최적화할 수 있었는지 분석한다.

**스코프 초과 달성의 의미**: 원래 Phase 12(SEO + 배포)가 MVP 완료 기준이었으나, Phase 13~16(Canonical URL, H1 소개, Footer, 검색)까지 추가로 달성했다. 이는 하네스 기반 자동화가 후반부 작업 속도를 가속시킨 결과다.

**주목할 패턴**: Phase 8과 Phase 5는 각각 "레이아웃 변경"과 "새 라이브러리 통합"이라는 높은 불확실성 작업이다. 불확실성이 높을수록 커밋 수가 증가하는 경향을 보인다.


**크리티컬 패스**: Phase 0-1 → 4 → 5 → 6-7 → 8 → 12 → 16

이 경로의 총 커밋 수는 약 180개로, 전체 267개의 67%를 차지한다. 크리티컬 패스 위의 작업이 지연되면 전체 프로젝트 일정에 직접적 영향을 준다.

**병렬 가능했던 구간**:

| 병렬 그룹 | 구성 | 절약 가능 시간 |
|-----------|------|---------------|
| A | Phase 2 (Git) ↔ Phase 3-A (디자인 기획) | ~0.25일 |
| B | Phase 9 (접근성) ↔ Phase 10 (SEO) ↔ Phase 11 (배포) | ~0.5일 |
| C | Phase 13 (Canonical) ↔ Phase 14 (H1) ↔ Phase 15 (Footer) | ~0.5일 |

**실제로는 병렬화되지 않은 이유**: 1인 개발이므로 물리적으로 동시 작업 불가. 단, AI 협업에서 에이전트 병렬 실행(worktree isolation)을 활용했다면 그룹 B, C는 병렬 가능했다.

### 4. "처음부터 다시 한다면" 최적 Phase 순서

#### 실제 진행 vs 이상적 순서

```
실제 진행 (7일):
Day 1: Phase 0-3B (인프라 + 디자인)           56커밋
Day 2: Phase 4-5 기획                         14커밋
Day 3: Phase 5 (3D 그래프)                    46커밋
Day 4: (공백)                                  0커밋
Day 5: Phase 6-8 (분할 뷰 + 리디자인)          37커밋
Day 6: Phase 8-12 (리디자인 마무리 + 배포)      59커밋
Day 7: Phase 13-16 + 회고                      55커밋

이상적 재설계 (6일):
Day 1: Phase 0-1 (초기화) + Phase 3-A (디자인 기획)     동시 진행
Day 2: Phase 3-B (토큰) + Phase 4 (콘텐츠 파이프라인)   순차 필수
Day 3: Phase 5 (3D 그래프)                              탐색적 — 하루 확보
Day 4: Phase 6-8 (분할 뷰 + 리디자인)                   핵심 UI 집중
Day 5: Phase 9 ↔ Phase 10-11 (접근성 + SEO + 배포)     병렬 실행
Day 6: Phase 12-16 (OG + 완성도)                        병렬 실행
```

#### 실제와 이상적 순서의 차이점

| 차이 | 실제 | 이상적 | 이유 |
|------|------|--------|------|
| Day 4 공백 | 0커밋 (휴식/검토) | 제거 — Day 3에 통합 | 리듬 유지가 생산성에 유리 |
| Phase 2 독립 실행 | Phase 1 후 순차 | Phase 3-A와 병렬 | Git 설정은 디자인 기획과 무관 |
| Phase 9-11 순차 | Day 6에 몰아서 | Day 5에 병렬 실행 | 접근성/SEO/배포는 상호 독립 |
| Phase 5 기획 분리 | Day 2에 별도 | Day 2 후반부에 통합 | 기획과 구현 사이 간격 최소화 |

#### 핵심 개선 포인트

**1) Day 1 과부하 방지**: 실제 Day 1에 56커밋이 몰렸다. 초기화 + 디자인 + 하네스를 동시에 진행한 결과다. 이상적으로는 Day 1을 "초기화 + 기획"으로 제한하고, 구현은 Day 2부터 시작하는 것이 지속 가능하다.

**2) 탐색적 Phase에 버퍼 확보**: Phase 5(3D 그래프)는 기술 불확실성이 가장 높았다. 실제로는 Day 2~3에 걸쳐 진행했지만, 이상적으로는 하루를 온전히 확보하고 라이브러리 탐색 → 프로토타입 → 본 구현 순으로 집중하는 것이 효율적이다.

**3) 후반부 병렬화**: Phase 9~16은 상호 독립적인 작업이 많다. 실제로는 순차 진행했지만, worktree 기반 병렬 에이전트를 활용하면 Day 6-7을 Day 5-6으로 압축 가능하다.

| 하네스 기능 | 절약 효과 | 추정 절약 시간/일 |

**일일 절약 추정**: ~90분/일 × 6일(Day 2~7) = **~9시간 절약**

#### 하네스 없이 진행했다면?

```
하네스 투자 시간:   ~8시간 (Day 1 대부분)
하네스 절약 시간:   ~9시간 (Day 2~7 누적)
순수 ROI:          +1시간 (7일 기준)

하지만 8일 이상이면:
  8일:  +2.5시간
  14일: +12.5시간
  21일: +22.5시간
```

**핵심 인사이트**: 7일 프로젝트에서 하네스 ROI는 거의 손익분기점이다. 하네스의 진짜 가치는 **시간 절약이 아니라 품질 일관성**에 있다. PostToolUse 훅이 매 편집마다 타입 검사를 실행함으로써 Day 6-7의 fix 커밋 수를 줄였고, 배포 직전 긴급 수정이 필요 없었다.

## 결론

### 핵심 답변: 7일 규모가 적절했는가?

**적절했다.** 근거:

1. **MVP 완료 + 추가 기능**: Phase 12(MVP)까지가 원래 목표였으나 Phase 16까지 달성했다. 7일이 부족하지 않았다는 증거다.
2. **Day 4 공백이 증명하는 것**: 개발 중간에 하루 쉴 수 있었다는 것은 일정에 여유가 있었다는 의미다. 무리한 스코프였다면 공백 없이 달렸을 것이다.
3. **커밋 품질 유지**: fix 비율이 10%로 안정적이었다. 급하게 밀어붙이면 fix 비율이 20% 이상으로 올라가는 것이 일반적이다.
4. **후반부 가속**: Day 6(59커밋)과 Day 7(55커밋)이 최고점이었다. 하네스와 패턴이 안정화되면서 후반부로 갈수록 빨라졌다.

단, **하네스 구축을 뺐다면 5일도 가능했을 것**이다. 하네스 없이 5일 완성 vs 하네스 있이 7일 완성 — 이 프로젝트만 놓고 보면 비슷한 결과다. 하네스의 진짜 가치는 **다음 프로젝트에서 Day 1부터 생산성이 높아지는 것**이다.

### Phase 순서 최적화 핵심 교훈

1. **불확실성이 높은 Phase를 앞으로**: Phase 5(3D 그래프)는 "될지 안 될지 모르는" 작업이었다. 이런 작업이 후반에 배치되면 전체 일정이 위험해진다. 실제로 Day 2~3에 배치한 것은 올바른 판단이었다.

2. **기획과 구현 사이 간격을 최소화**: Phase 5-0(기획)과 Phase 5(구현) 사이에 하루 간격이 있었다. 기획 직후 바로 구현에 들어가는 것이 컨텍스트 유지에 유리하다.

3. **후반부는 병렬화 설계**: Phase 9~16은 상호 독립적인 "완성도 개선" 작업들이다. 이런 작업은 설계 시점에 병렬 실행 가능하도록 의존성을 끊어두면 일정 단축이 가능하다.

### 다음 프로젝트에 적용할 Phase 설계 원칙

```
Day 0: 스코프 확정 + 의존성 DAG 작성
Day 1: 초기화 + 하네스 핵심 (CLAUDE.md + 훅 4개 + 스킬 3개)
Day 2: 가장 불확실한 기술 탐색 (프로토타입)
Day 3~N-2: 크리티컬 패스 순서로 구현
Day N-1: 독립적 완성도 작업 (병렬 실행)
Day N: 배포 + 회고
```

**Phase 크기 기준**: 하루에 2~3 Phase가 적정. Phase 하나가 30커밋을 넘으면 분할을 고려한다 (Phase 8의 55커밋은 2~3개로 나눌 수 있었다).
