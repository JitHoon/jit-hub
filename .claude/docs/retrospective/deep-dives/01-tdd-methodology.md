## 배경

대시보드 리뷰 포인트 #4: "테스트 주도 개발을 어떻게 하는지 잘 모르겠다. 기획 및 UI/UX 의도에 따라서 테스트 코드를 먼저 짜고 기능 구현을 해야하는지, 아니면 기능 구현에 맞게 테스트 코드를 짜야하는지 잘 모르겠다."

이 프로젝트에서 유닛 테스트 5개, E2E 테스트 7개를 작성했지만, TDD 방법론에 대한 체계적 이해 없이 진행했다. 이 심화에서는 TDD의 기본 원리를 학습하고, 이 프로젝트에 적용했다면 어떤 순서였을지 시뮬레이션한다.

## 분석 / 학습 내용

### 1. TDD 핵심 사이클: Red-Green-Refactor

TDD는 세 단계를 짧은 주기로 반복하는 개발 방법론이다.

```
Red   → 실패하는 테스트를 먼저 작성한다 (아직 구현이 없으므로 실패)
Green → 테스트를 통과시키는 최소한의 코드를 작성한다
Refactor → 통과된 코드를 정리한다 (테스트는 계속 통과해야 함)
```

핵심은 **"실패하는 테스트가 먼저 존재해야 코드를 쓸 수 있다"**는 규율이다. 이 규율이 주는 이점:

- 구현 전에 "무엇을 만들어야 하는가"를 명확히 정의한다
- 테스트 없는 코드가 생기지 않는다
- 과도한 구현(YAGNI 위반)을 방지한다

### 2. "테스트 먼저" vs "기능 먼저" — 상황별 판단 기준

둘 다 정당한 접근법이며, 상황에 따라 선택한다.

#### 테스트 먼저가 유리한 경우

| 조건 | 이유 | 프로젝트 예시 |
|------|------|-------------|
| 입출력 규칙이 명확할 때 | 테스트를 먼저 정의할 수 있음 | `schema.test.ts` — Zod 스키마 검증 규칙 |
| 엣지 케이스가 핵심일 때 | 실패 사례를 먼저 정의해야 누락 방지 | `fix-emphasis.test.ts` — 한글/구두점 처리 |
| 데이터 계약(contract)을 정의할 때 | 인터페이스를 테스트로 먼저 고정 | `node-connections.test.ts` — 연결 맵 구조 |
| UI/UX 요구사항이 구체적 수치일 때 | "다크모드 배경 #111111" 같은 스펙 | `theme.spec.ts` — 색상값, localStorage 유지 |

#### 기능 먼저가 유리한 경우

| 조건 | 이유 | 프로젝트 예시 |
|------|------|-------------|
| 탐색적으로 프로토타이핑할 때 | 정답을 모르는 상태에서 테스트 정의 불가 | 3D 그래프 초기 렌더링 실험 |
| 외부 라이브러리 API를 학습할 때 | 동작을 먼저 확인해야 테스트 가능 | react-force-graph-2d 통합 |
| 상수/설정 데이터를 정의할 때 | 검증 대상이 고정값이므로 TDD 가치 낮음 | `cluster.test.ts` — 클러스터 상수 |
| 시각적 결과가 판단 기준일 때 | "보기 좋은가"는 assertion으로 표현 불가 | 그래프 레이아웃 조정 |

**핵심 판단 기준**: "테스트의 기대값(expected)을 구현 전에 알 수 있는가?"
- **알 수 있다** → 테스트 먼저 (TDD)
- **알 수 없다** → 기능 먼저 → 동작 확인 → 테스트 추가 (TAD, Test After Development)

### 3. BDD: 기획/UI/UX 의도에서 테스트로

BDD(Behavior-Driven Development)는 TDD의 확장으로, **사용자 행동 시나리오**에서 테스트를 도출한다.

```
Given [초기 상태]
When  [사용자 행동]
Then  [기대 결과]
```

이 프로젝트의 E2E 테스트는 이미 BDD에 가까운 구조를 가지고 있다:

```typescript
// theme.spec.ts — 실제 코드 패턴
test('테마 토글 → 다크모드 전환', async ({ page }) => {
  // Given: 페이지 로딩 (초기 상태)
  await page.goto('/');
  
  // When: 테마 토글 클릭 (사용자 행동)
  await page.click('[data-testid="theme-toggle"]');
  
  // Then: 배경색이 #111111 (기대 결과)
  expect(bgHex).toBe('#111111');
});
```

**기획 문서 → BDD 테스트 도출 과정**:

```
기획: "사용자가 테마를 전환하면 배경색이 바뀌고, 새로고침해도 유지된다"
    ↓
시나리오 1: 토글 클릭 → 다크모드 배경 #111111
시나리오 2: 토글 클릭 → 라이트모드 배경 #F7F7F7
시나리오 3: 다크모드 선택 → 새로고침 → 여전히 다크모드
시나리오 4: localStorage에 테마 값 저장됨
    ↓
E2E 테스트 4개 작성 (Red)
    ↓
테마 시스템 구현 (Green)
    ↓
CSS 변수 정리 (Refactor)
```

### 4. 프로젝트 테스트 TDD 시뮬레이션

이 프로젝트의 실제 기능 5개에 대해, TDD로 진행했다면 어떤 순서였을지 시뮬레이션한다.

#### 시뮬레이션 A: Zod 프론트매터 스키마 (TDD 적합도: ★★★)

git 히스토리에서 스키마는 기능 구현과 함께 만들어졌다. TDD였다면:

```
[Red 1] "slug 필드가 없으면 파싱 실패해야 한다" → 실패
[Green 1] slug를 z.string()으로 정의
[Red 2] "cluster가 유효한 ID가 아니면 실패해야 한다" → 실패
[Green 2] cluster를 z.enum(CLUSTER_IDS)로 정의
[Red 3] "summary 미입력 시 빈 문자열 기본값" → 실패
[Green 3] summary를 z.string().default('')로 정의
[Refactor] 타입 추출, ParsedNode 인터페이스 정리
```

**효과**: 스키마 필드를 하나씩 추가하면서 각 필드의 존재 이유가 테스트로 문서화됨.

#### 시뮬레이션 B: 노드 연결 맵 (TDD 적합도: ★★★)

```
[Red 1] "A→B 연결이 있으면 A의 연결 목록에 B가 포함된다" → 실패
[Green 1] 기본 맵 생성 로직
[Red 2] "A→B이면 B의 연결 목록에도 A가 포함된다 (양방향)" → 실패
[Green 2] 양방향 연결 로직 추가
[Red 3] "A→B가 두 번 나와도 중복 없이 1번만" → 실패
[Green 3] Set 기반 중복 제거
[Red 4] "relationship 필드가 있으면 메타데이터에 포함" → 실패
[Green 4] relationship 파싱 로직
[Refactor] Map<string, ConnectionInfo[]> 타입 정리
```

**효과**: 양방향 연결, 중복 제거 같은 까다로운 로직을 한 단계씩 안전하게 구축.

#### 시뮬레이션 C: 테마 전환 E2E (TDD 적합도: ★★☆)

```
[Red 1] "토글 클릭 → html에 dark 클래스 추가" → 실패 (아직 토글 없음)
[Green 1] ThemeToggle 컴포넌트 + 기본 전환 로직
[Red 2] "다크모드 배경 #111111" → 실패 (CSS 미적용)
[Green 2] CSS 변수 설정
[Red 3] "새로고침 후에도 유지" → 실패 (localStorage 없음)
[Green 3] localStorage 읽기/쓰기 추가
[Refactor] useTheme 훅으로 로직 추출
```

**주의**: E2E 테스트는 실행 비용이 높으므로, 모든 단계에서 E2E를 돌리기보다 유닛 테스트(useTheme 훅)로 빠르게 Red-Green 사이클을 돌리고, 최종 통합만 E2E로 검증하는 것이 실용적이다.

#### 시뮬레이션 D: 3D 그래프 렌더링 (TDD 적합도: ★☆☆)

```
실제 개발: react-force-graph-2d 탐색 → 동작 확인 → 컨테이너 렌더링 → WebGL 폴백 추가
TDD 시도: "canvas가 렌더링되어야 한다" → ... 어떤 canvas? 어떤 크기? 라이브러리 API를 모름
```

**결론**: 외부 라이브러리 통합은 탐색적 개발(기능 먼저) 후, 동작이 확정되면 회귀 방지 테스트를 추가하는 것이 현실적이다. 이 프로젝트의 `home-3d.spec.ts`가 정확히 이 패턴을 따랐다.

#### 시뮬레이션 E: 마크다운 강조 수정 (TDD 적합도: ★★★)

```
[Red 1] "**bold** → <strong>bold</strong>" → 실패
[Green 1] 기본 정규표현식 변환
[Red 2] "한글**굵게**한글 → 한글 사이도 변환" → 실패
[Green 2] 유니코드 범위 포함하도록 정규표현식 확장
[Red 3] "코드블록 내 **text**는 변환하지 않음" → 실패
[Green 3] 코드블록 예외 처리
[Refactor] 정규표현식 가독성 개선
```

**효과**: 텍스트 변환의 엣지 케이스를 사전에 정의하여 "한글 옆의 강조가 깨진다" 같은 버그를 원천 방지.

### 5. TDD의 현실적 적용 전략

모든 코드에 TDD를 적용하는 것은 비현실적이다. 이 프로젝트의 경험에서 도출한 전략:

```
┌─────────────────────────────────────────────────┐
│         TDD 적용 스펙트럼 (이 프로젝트 기준)        │
├─────────────────────────────────────────────────┤
│                                                 │
│  항상 TDD        조건부 TDD        TAD 권장       │
│  ◄────────────────┼────────────────►           │
│                                                 │
│  - Zod 스키마      - React 훅       - 3D 렌더링  │
│  - 순수 유틸       - API 핸들러     - 라이브러리   │
│  - 데이터 변환     - 상태 관리        탐색        │
│  - 정규표현식      - E2E 시나리오   - 시각 디자인  │
│                                                 │
│  기대값을 미리     기대값을 대략      기대값을 구현 │
│  정확히 안다       알지만 세부 불확실  후에야 안다   │
└─────────────────────────────────────────────────┘
```

## 결론

### 핵심 답변: "기획 기반 테스트 먼저" vs "기능 기반 테스트 나중"

둘 중 하나를 선택하는 문제가 아니라, **코드의 성격에 따라 다르게 적용**하는 것이 정답이다.

| 판단 기준 | 접근법 |
|----------|--------|
| 기대값을 구현 전에 알 수 있는가? | **Yes** → TDD (테스트 먼저) |
| 외부 라이브러리를 탐색 중인가? | **Yes** → TAD (기능 먼저, 테스트 나중) |
| 기획서에 구체적 수치가 있는가? | **Yes** → BDD 스타일 E2E 먼저 |
| 순수 함수인가? | **Yes** → TDD가 가장 효과적 |

### 이 프로젝트에서 배운 것

1. **schema.test.ts, fix-emphasis.test.ts, node-connections.test.ts**는 TDD로 작성했으면 더 체계적이었을 가능성이 높다 — 입출력이 명확한 순수 함수들이다.
2. **theme.spec.ts**는 기획서의 색상값(#111111, #F7F7F7)이 명확했으므로 BDD 스타일 TDD가 가능했다.
3. **3D 그래프 렌더링**은 TAD가 현실적이었다 — 라이브러리 탐색이 선행되어야 했다.
4. **테스트 피라미드의 역삼각형**(E2E 7 > Unit 5)은 TDD 부재의 결과이기도 하다 — TDD를 하면 자연스럽게 유닛 테스트가 먼저 쌓인다.

### 다음 기능 개발에 적용할 실천 규칙

1. **새 유틸 함수 작성 시**: 빈 함수 선언 → 테스트 3개 이상 먼저 작성 → 구현
2. **Zod 스키마 수정 시**: 실패 케이스 테스트 먼저 → 스키마 수정
3. **UI 기능 구현 시**: 기획서에서 Given/When/Then 추출 → E2E 작성 → 구현
4. **라이브러리 탐색 시**: 프로토타입 먼저 → 동작 확인 → 회귀 테스트 추가
