# 다음 프로젝트 킥오프 인터뷰 가이드

> 새 프로젝트를 시작할 때, 과거 시행착오에서 배운 모든 결정사항을 인터뷰 형식으로 빠르게 확정하는 프레임워크.
> `/plan-product` 실행 전에 이 가이드를 먼저 완료하여 프로젝트 기반 결정을 마친다.

---

## 사용법

1. 각 섹션의 **인터뷰 질문**을 순서대로 진행한다
2. 질문마다 **선택지**와 **권장안**(과거 근거 포함)이 제시된다
3. 결정이 끝나면 **결정 체크리스트**를 체크한다
4. 모든 섹션 완료 후 결정사항을 프로젝트 blueprint 문서에 반영한다

---

## 3. 아키텍처 & 기술 스택

### 배경
> jit-hub에서 8개 주요 기술 결정을 내렸고 번복은 0건 (v1→v2는 "진화"로 분류).
> 핵심 성공 요인: 기술 선택 시 "왜 이것이 아닌가"를 먼저 검토한 소거법.
> feature-first 디렉토리 구조가 유지보수성 9/10 달성의 핵심이었다.

### 인터뷰 질문

**Q3-1. 프레임워크 선택은?**
- 선택 기준: 렌더링 요구사항(SSG/SSR/CSR), 배포 환경, 생태계 성숙도
- 대안 검토: 각 후보의 장단점을 "왜 이것이 아닌가" 관점으로 소거
- 권장: 결정 근거를 blueprint에 기록 (나중에 되돌아볼 수 있도록)

**Q3-2. 렌더링 전략은?**

| 전략 | 적합한 경우 |
|------|------------|
| SSG (정적 생성) | 콘텐츠 중심, SEO 중요 |
| SSR (서버 렌더링) | 사용자별 동적 콘텐츠 |
| CSR (클라이언트) | 인터랙티브 앱, SEO 불필요 |
| 하이브리드 | SEO + 인터랙션 동시 필요 (jit-hub 채택) |

- 권장: 하이브리드 시 듀얼 라우트 전략 고려 (UX용 + SEO용 분리)

**Q3-3. 타입 안전성 수준은?**

| 수준 | 설정 |
|------|------|
| 기본 | TypeScript strict |
| 강화 (권장) | + noUncheckedIndexedAccess |
| 최대 | + Zod 런타임 검증 (외부 데이터) |

- 권장: "강화" 이상. `any` 금지는 절대 규칙으로 설정
- 근거: jit-hub에서 Type Safety 9/10 달성의 핵심

**Q3-4. 디렉토리 구조는?**
- 권장: feature-first colocalization
  ```
  src/features/{feature-name}/  → types, utils, components, hooks
  src/components/               → 공유 UI
  src/lib/                      → 순수 유틸리티
  src/types/                    → 공유 타입
  ```
- 의존성 방향: shared → features → app (단방향)
- 크로스 피처 임포트 금지 (필요 시 shared로 승격)

**Q3-5. 상태 관리 전략은?**
- A) URL 상태 (검색 파라미터) — 공유 가능, 북마크 가능
- B) React 상태 (useState/useReducer) — 컴포넌트 로컬
- C) 외부 스토어 (useSyncExternalStore) — 테마, 전역 설정
- D) 서버 상태 (React Query 등) — API 연동 시
- 권장: URL 상태 + 외부 스토어 조합 (서버 상태 라이브러리는 필요할 때만)

**Q3-6. 코드 구조화 원칙은?**

| 원칙 | 내용 | 근거 |
|------|------|------|
| 공유 유틸 배치 기준 | 2개+ 피처에서 사용 → `src/lib/`, 단일 피처 → `src/features/{f}/utils/` | retro #31에서 추출 위치 결정이 리팩토링 핵심 |
| 순수 함수 추출 우선 | useCallback 래핑보다 순수 함수 추출이 테스트 가능성 + 재사용성 모두 우수 | 65개 유닛 테스트가 순수 함수 중심으로 작성됨 |
| 코로케이션 패턴 | page 전용 컴포넌트는 `_components/` 하위에 배치 (Next.js 컨벤션) | 라우트와 컴포넌트의 관계를 디렉토리로 표현 |
| ref 통합 | 여러 개의 useRef → 단일 ConfigSnapshot 객체로 통합 | 렌더링 간 일관성 + 동기화 useEffect 감소 |

- 권장: 데이터/계산/액션 분리 원칙 적용 (Eric Normand)
  - **데이터**: 불변 값, 타입 정의
  - **계산**: 순수 함수 (입력→출력, 부수효과 없음)
  - **액션**: 외부 상태 변경 (API 호출, DOM 조작, 이벤트 핸들러)

**Q4-2. 콘텐츠 관리 방식은?**

| 방식 | 장점 | 단점 | 적합 |
|------|------|------|------|
| 헤드리스 CMS | 비개발자 편집 가능 | 외부 의존성, 비용 | 팀 협업, 비개발자 편집 |

**Q4-3. 스키마 검증은 어디서?**

**Q4-4. 단일 소스 원칙을 적용할 수 있는가?**
- "콘텐츠 추가 = 파일 하나 생성"이 가능한 구조인가?
- 권장: 가능하다면 반드시 적용. 다른 파일 수정 불필요한 구조가 이상적
- 근거: jit-hub의 "새 노드 = .md 파일 하나" 원칙이 유지보수 핵심

| 방식 | 적합 | 구성 |
|------|------|------|
| 없음 | XS | 부채 발생 전에 프로젝트 종료 |
| 인라인 TODO + 종료 시 일괄 정리 | S | TODO 주석 → 회고 시 수집 |
| 부채 레지스터 (별도 문서) | M (권장) | 의도적/비의도적 분류 + 우선순위 |

- 권장: S 이상에서 부채를 **의도적**(전략적 스코프 결정) vs **비의도적**(주의 부족)으로 분류
- 근거: 의도적 부채는 수용, 비의도적 부채는 즉시 해결 대상 — 구분 없이 쌓으면 관리 불가

**Q11-2. 부채 해결 타임라인은?**

| 크기 | 시간 | 적합한 작업 |
|------|------|------------|
| XS | 30분 이내 | 중복 파일 삭제, 미사용 코드 제거, 설정 수정 |
| S | 1-2일 | 테스트 보강, 성능 최적화, 도구 도입 |
| M | 1주 | 컴포넌트 분해, 대규모 리팩토링 |

- 권장: XS부터 즉시 실행 → 작은 성공이 모멘텀 제공
- 근거: jit-hub 31개 액션 중 XS 10건을 먼저 처리하여 실행 리듬 확립

**Q11-3. 대규모 리팩토링 시 배치 전략은?**
- 100줄 이상 변경 시: 독립 배치로 분할, 각 배치 후 lint + test 검증
- 실행 순서: 의존성 없는 배치 먼저, 의존성 있는 배치 나중에
- 각 배치의 검증 기준을 사전에 정의 (컴파일 통과 / 테스트 통과 / lint 통과)
- 근거: retro #31에서 ~327줄을 5배치로 분할, 순서 최적화(1→3→2→4→5)로 안전하게 완료

**Q11-4. 회고에서 액션 도출 프로세스는?**
- 권장 사이클: 회고 → 인사이트 요약 → 액션 도출 → XS/S/M 분류 → 체계적 실행
- 실행 도구: 액션 테이블 + 상태 컬럼(`[ ]`/`[x]`) + 원본 참조
- 근거: `/retrospective` 커맨드로 하나씩 체크하며 진행하는 패턴이 31건 완주에 효과적
