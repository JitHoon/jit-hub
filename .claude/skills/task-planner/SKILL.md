# 태스크 플래너 Skill

## 태스크 크기 기준

| 크기 | 정의 | 판단 기준 |
|------|------|---------|
| **XS** | 1 파일, 1 함수, 1 타입 블록 | "방금 머릿속에 그린 대로 바로 쓸 수 있다" |
| **S** | 1 컴포넌트 또는 1 훅 (로직 포함) | "설계가 끝나면 30분~2시간 안에 완성할 수 있다" |
| **M** | 컴포넌트 + 훅 + 통합 묶음 | "반나절이면 충분하다" |
| **L** | 하루 이상 | **progress.md에 쓰지 않는다** — 반드시 쪼갠다 |

## 좋은 태스크의 기준

다음 3가지를 모두 만족하면 올바른 크기다:

1. **독립 커밋 가능** — 이 태스크만 완료해도 혼자 커밋할 수 있다
2. **완료 기준 한 줄** — "이것이 되면 완료다"를 한 줄로 쓸 수 있다
3. **수정 파일 3개 이하** — (`index.ts` re-export 제외)

## 태스크 도출 사고 순서

기획 문서를 읽은 뒤 다음 순서로 생각한다:

1. **TECH_PLAN 신규 파일 목록**을 본다 — 각 파일이 태스크 후보다
2. **파일 간 의존성**을 파악한다 — 어떤 파일이 먼저 있어야 다른 파일을 쓸 수 있는가?
3. **PRD 완료 기준**과 대조한다 — 파일 구현만으로 PRD 기준을 충족하는가? 통합이 필요하면 별도 태스크
4. **DESIGN 컴포넌트 계층**을 확인한다 — 하위 컴포넌트 먼저, 상위 조립 나중
5. **ROADMAP MVP 범위**로 범위를 확정한다 — Phase 2 이후 항목은 이번 Phase에 넣지 않는다

## progress.md 포맷 규칙

### Phase 번호

기존 progress.md에서 가장 큰 Phase 번호 확인 → +1부터 시작.

### Phase 헤더

```markdown
### Phase [N]: [기능명] · 브랜치: `feat/[feature-slug]`
```

### 태스크 행

```markdown
| [N]-[순번] | [동사로 시작하는 작업 설명] | [XS/S/M] | [ ] |
<!-- 완료 기준: [한 줄] -->  ← S 이상 태스크만
```

동사 예: 설치, 정의, 구현, 추가, 연결, 조립, 검증

### 상태 표기

`[ ]` 미완료 / `[x]` 완료 / `[~]` 진행 중

## Gotchas

- "및", "그리고", "+" 포함된 태스크 설명은 분리 신호다
- ROADMAP Phase(MVP/Phase2/Phase3)를 progress.md Phase 번호로 착각하지 않는다
- 의존성 순서는 태스크 번호 순서로 표현한다 — 별도 표기 불필요
- 기존 progress.md의 완료된 Phase와 "핵심 결정 사항"은 수정하지 않는다
