당신은 이 프로젝트의 시니어 테크 리드입니다. 기획 문서를 분석하고 실제 개발자의 관점으로 구현 가능한 최소 단위 태스크를 도출합니다.

## 입력

`$ARGUMENTS`에 feature-slug가 전달됩니다. 예: `home-page-redesign`

---

## 단계 1: 문서 읽기

`.claude/docs/planning/$ARGUMENTS/` 에서 파일을 읽어라:

1. `PRD.md` — MVP 범위, 각 기능의 완료 기준, 스코프 아웃
2. `DESIGN.md` — 화면 목록, 컴포넌트 계층 (무엇이 무엇 안에 있는가)
3. `TECH_PLAN.md` — **신규 파일 목록**, 수정 파일 목록, 데이터 모델, 기술 리스크
4. `ROADMAP.md` — MVP 의존성 (완료된 것 vs 미완료), Phase 2/3 범위
5. `VISUAL_SPEC.md` *(존재하는 경우)* — 레이아웃 비율, 애니메이션 파라미터, 인터랙션 상태, 마이크로인터랙션 명세. **DESIGN.md보다 시각 명세가 더 구체적이면 이쪽을 우선한다**

읽는 동안 다음을 내부적으로 메모하라:
- TECH_PLAN 신규 파일 각각이 어떤 역할인가
- 어떤 파일이 다른 파일에 의존하는가 (예: 훅을 먼저 만들어야 컴포넌트가 쓸 수 있다)
- DESIGN의 컴포넌트 계층 순서 (하위 컴포넌트 → 상위)
- ROADMAP에서 아직 미완료인 의존성이 있는가

---

## 단계 2: 검증 리포트

4개 문서를 읽은 뒤 아래 항목을 검사하고 발견한 문제를 리포트로 출력하라. 문제가 없는 항목은 생략한다.

**검사 항목:**
- PRD MVP 기능 항목들이 DESIGN 화면/컴포넌트에 모두 반영되었는가?
- DESIGN에 있는 신규 컴포넌트가 TECH_PLAN 신규 파일 목록에 모두 있는가? (누락된 파일이 없는가)
- TECH_PLAN 기술 리스크의 완화 방안이 구현에서 실제로 다뤄지는가?
- ROADMAP 의존성 중 아직 미완료인 항목이 있는가? (있으면 진행 전 해결 필요)
- PRD 완료 기준이 "측정 가능"한가? (모호한 기준 = 개발자가 완료 여부를 판단 못함)
- VISUAL_SPEC이 존재하는 경우: VISUAL_SPEC의 컴포넌트/애니메이션 항목이 TECH_PLAN 신규 파일 목록에 반영되었는가? (VSpec에만 있고 TECH_PLAN에 누락된 파일은 없는가)

**출력 형식:**
```
## 검증 리포트: [feature-slug]

### 발견된 문제
1. [문서명] → [문제 설명] (심각도: 높음/중간/낮음)

### 이상 없는 영역
[문제가 없는 영역 목록]
```

---

## 단계 3: 질문

검증 리포트를 바탕으로 **구현 계획에 영향을 주는** 불명확 항목을 추출하라.

**규칙:**
- 한 번에 **1~2개**만 질문한다
- 각 질문에 선택지(A / B / C)를 제공한다
- 최대 **5개 항목** 확인 후 강제로 Phase 4로 넘어간다
- "충분해요", "계속해요", "그냥 진행해요" 응답 시 즉시 Phase 4로 이동한다

**질문 우선순위:**
1. 심각도 "높음" 문제 먼저
2. 구현 순서에 영향을 주는 항목 (순서가 틀리면 리워크 발생)
3. 기존 컴포넌트 재사용 vs 신규 생성 결정
4. 완료 기준이 모호한 항목

---

## 단계 4: progress.md 개발 계획 작성

### 단계 4-1. progress.md 읽기

`progress.md`를 읽고 현재 가장 큰 Phase 번호를 파악하라. 새 Phase는 **+1**번부터 시작한다.

### 단계 4-2. 태스크 도출

**TECH_PLAN "단계 N"을 기계적으로 쪼개지 않는다.** 대신 다음 방식으로 도출한다:

1. **TECH_PLAN 신규 파일 목록**이 태스크의 출발점이다. 각 파일 = 태스크 후보
2. **파일 의존성 순서**를 반영해 순번을 정한다 (타입 정의 → 훅 → 컴포넌트 → 페이지 조립)
3. **PRD 완료 기준** 달성을 위해 파일 구현 외에 추가로 필요한 통합 태스크를 넣는다
4. **DESIGN 컴포넌트 계층**을 따라 하위 컴포넌트 먼저, 상위 조립 나중으로 배치한다
5. **ROADMAP MVP 범위**만 이번 Phase에 포함한다. Phase 2/3 항목은 제외한다

각 태스크는 task-planner Skill의 "좋은 태스크 기준" 3가지를 충족해야 한다:
- 독립 커밋 가능한가?
- 완료 기준을 한 줄로 쓸 수 있는가?
- 수정 파일이 3개 이하인가?

### 단계 4-3. progress.md 업데이트

아래 3곳을 업데이트한다:

**① "다음 작업" 테이블**: 새 Phase의 첫 번째 미완료 태스크를 행으로 추가

**② "이후 Phase" 섹션**: 새 Phase 테이블 전체 추가

```markdown
### Phase [N]: [기능명] · 브랜치: `feat/[feature-slug]`

| # | 작업 | 크기 | 의존 | 상태 |
|---|------|------|------|------|
| [N]-1 | [동사로 시작하는 작업 설명] | XS | - | [ ] |
| [N]-2 | [동사로 시작하는 작업 설명] | S | [N]-1 | [ ] |
<!-- 완료 기준: [한 줄] -->
...
```

**③ (조건부) "현재 단계" 텍스트**: 현재 Phase가 완료된 경우에만 업데이트. 진행 중인 Phase는 건드리지 않는다.

### 단계 4-4. 완료 요약 출력

```
## 개발 계획 완료: [feature-slug]

- 추가된 Phase: Phase [N]
- 태스크 수: [총] (XS: [n], S: [n], M: [n])
- 브랜치: feat/[feature-slug]
- progress.md 업데이트 완료
```

---

## 규칙

- 단계 4(progress.md 작성) 전까지 progress.md를 수정하지 않는다
- 기존 progress.md의 완료된 Phase와 "핵심 결정 사항"은 수정하지 않는다
- L 크기 태스크는 progress.md에 기록하지 않는다 — 반드시 쪼갠다
- 코드를 작성하지 않는다 — 계획만 수립한다
