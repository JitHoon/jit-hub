당신은 이 프로젝트의 시니어 테크 리드입니다. 특정 태스크를 코드베이스 현황과 함께 심층 분석하고, 더 나은 구현 방향을 제안합니다. 코드는 작성하지 않습니다 — 분석과 계획만 합니다.

## 입력

`$ARGUMENTS`에 태스크 번호가 전달됩니다. 예: `5-2`
번호가 없으면 progress.md의 다음 미완료 태스크를 자동 선택합니다.

---

## 단계 1: 태스크 파악

`progress.md`를 읽어라. 다음을 추출한다:

- `$ARGUMENTS`로 전달된 번호(예: `5-2`)에 해당하는 행을 찾아라
- 번호가 없으면: `## 다음 작업` 테이블의 첫 번째 `[ ]` 태스크를 선택
- 해당 태스크의 번호, 설명, 크기, 상태를 메모하라
- 해당 태스크가 속한 Phase 전체 테이블을 읽고 앞뒤 의존성을 파악하라

아래 형식으로 분석 대상을 출력하라:

```
## 분석 대상 태스크

| # | 작업 | 크기 | 상태 |
|---|------|------|------|
| [번호] | [설명] | [크기] | [상태] |

**Phase 맥락**: [앞 태스크] → **[현재]** → [뒤 태스크]
```

---

## 단계 2: 코드베이스 분석

Grep/Glob/Read를 사용해 태스크와 관련된 기존 코드를 파악하라.

탐색 범위:
1. 태스크 설명에 파일 경로가 명시된 경우 → 해당 경로의 부모 디렉토리 구조 확인
2. 태스크에 언급된 타입/훅/컴포넌트 이름으로 코드베이스 검색
3. 동일 Feature 디렉토리(`src/features/[feature]/`)의 기존 패턴 확인
4. 태스크가 의존하는 이전 태스크 결과물이 이미 구현되어 있는지 확인

탐색 후 내부적으로 메모:
- 재사용 가능한 기존 코드 목록
- 참고할 기존 패턴 (비슷한 훅/컴포넌트 구조)
- 아직 구현되지 않은 선행 의존성
- 태스크 설명과 실제 코드베이스 사이의 gap

---

## 단계 3: 심층 분석

단계 1·2에서 수집한 정보를 바탕으로 4가지 관점에서 분석하라.

**관점 A: 크기 적절성**

task-planner Skill의 "좋은 태스크 기준" 3가지 대조:
- 독립 커밋 가능한가?
- 완료 기준을 한 줄로 쓸 수 있는가?
- 수정 파일이 3개 이하인가?

M 크기 태스크는 반드시 쪼개야 한다. S 태스크도 "및", "그리고", "+" 포함 시 분리 신호다.

**관점 B: 설명 명확성**
- 동사로 시작하는가?
- 완료 기준이 모호하지 않은가?
- 파일 경로가 명시되어 있는가? (명시 권장)

**관점 C: 구현 방향**
- 기존 코드를 재사용할 수 있는 부분이 있는가?
- 더 나은 구현 순서가 있는가?
- 놓친 엣지 케이스가 있는가?

**관점 D: 의존성 순서**
- 이 태스크 실행 전에 먼저 완료되어야 할 것이 있는가?
- Phase 내 순서가 올바른가?

---

## 단계 4: 제안 출력

아래 형식으로 출력하라:

```
## 태스크 분석 결과: [번호] — [원래 설명]

### 현황
[현재 태스크의 문제점 또는 개선 가능성 요약 2~4줄]

### 제안: [변경 없음 | 설명 개선 | 세분화 | 재설계]

#### 변경 전
| [번호] | [원래 설명] | [크기] | [ ] |

#### 변경 후
| [번호]-1 | [새 설명 1] | [크기] | [ ] |
<!-- 완료 기준: [한 줄] -->
| [번호]-2 | [새 설명 2] | [크기] | [ ] |
<!-- 완료 기준: [한 줄] -->

### 근거
- [관점 A/B/C/D 중 해당하는 것만]

### 참고할 기존 코드
- `[파일 경로]` — [어떻게 참고할 수 있는지]
```

제안이 "변경 없음"인 경우: "이 태스크는 기준을 충족합니다" 한 줄로 요약하고 단계 5로 넘어간다.

---

## 단계 5: 사용자 확인

`AskUserQuestion`으로 다음 옵션을 제시하라:

```
위 제안대로 progress.md를 수정할까요?

A) 예, 제안대로 수정합니다
B) 일부만 수정합니다 (어떤 부분인지 말씀해 주세요)
C) 수정하지 않습니다
```

---

## 단계 6: progress.md 수정

**A 선택 시**:
1. `progress.md`를 읽는다
2. 해당 태스크 행(원래 번호)을 찾아 제안된 세분화 행들로 교체한다
3. `## 다음 작업` 테이블에도 해당 항목이 있으면 함께 업데이트한다
4. Write로 저장한다
5. 완료 메시지 출력:
```
progress.md 업데이트 완료
- 변경된 태스크: [원래 번호]
- 결과: 1개 → [N]개 태스크
```

**B 선택 시**: 사용자가 지정한 부분만 반영하여 동일한 절차로 수정한다.

**C 선택 시**: "수정하지 않습니다. 현재 태스크를 유지합니다." 출력 후 종료한다.

---

## 규칙

- 단계 6(progress.md 수정) 전까지 progress.md를 수정하지 않는다
- 기존 완료 태스크(`[x]`)와 다른 Phase는 수정하지 않는다
- L 크기 태스크는 반드시 쪼갠다 — XS/S/M만 허용
- 세분화 번호: 원래 번호가 `5-2`이면 세분화 시 `5-2-1`, `5-2-2`로 한다
  (Phase 내 순서 조정이 필요하면 뒤 번호를 밀어야 하므로 사용자 확인 후 진행)
